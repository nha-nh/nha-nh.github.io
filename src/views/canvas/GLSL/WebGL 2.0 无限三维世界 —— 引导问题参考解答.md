## 阶段零：全局架构与“无限世界”形式化定义

### Q0-1 活跃区块数与缓冲容量估算

> 若 R = 3、CHUNK_SIZE = 4，活跃区块数与最大实例数各是多少？实例 VBO 总容量是多少 MB？这个数字告诉你什么？

**解答：**

- 活跃区块数 = `(2R+1)³ = 7³ = 343` 个。
- 每区块最大实例数 = `CHUNK_SIZE³ = 4³ = 64` 个。
- 最大实例总数 = `343 × 64 = 21,952` 个。
- VBO 容量 = `21,952 × 64 B（一个 mat4）= 1,404,928 B ≈ 1.34 MB`。

这个数字告诉你三件事：

1. **内存完全不是问题**：1.34 MB 相对于现代 GPU 动辄数 GB 的显存可以忽略不计，整个“无限世界”的数据足迹是常数且极小。
2. **每帧上传带宽也不是问题**：即使每帧全量上传，60 FPS 下约 80 MB/s，远低于 PCIe / 统一内存的 GB/s 量级带宽。
3. **瓶颈在别处**：真正可能卡住的是顶点处理量（2.2 万实例 × 24 顶点 ≈ 53 万顶点/帧）、片元填充率（overdraw）和 CPU 端每帧的剔除/写入循环。这就是为什么指南后续花大量篇幅讲剔除而几乎不讲“省内存”。

### Q0-2 确定性生成的函数签名

> “确定性生成”意味着什么函数签名？它能接收任何随时间变化的参数吗？

**解答：**

伪签名应为：

```
generateChunk(cx: int, cy: int, cz: int, seed: int) → 实例矩阵列表
```

关键约束：**输出必须只由输入决定**（数学意义上的纯函数）。因此：

- **不能**接收：当前时间、摄像机位置、帧计数、`Math.random()` 的结果、任何可变全局状态。
- **可以**接收：世界种子 `seed` 这类“构造期固定、运行期不变”的配置——它本质上是函数定义的一部分而非变量。
- 实践形态上还会带一个 `outSlice: Float32Array` 输出参数（零分配要求），但这不影响“纯函数”语义：对同样的 (cx, cy, cz, seed)，写入 outSlice 的字节必须逐位一致。

判别测试：对同一坐标调用两次，输出逐字节相等；进程重启后再调用，输出仍然相等。第二条排除了“用 `Math.random()` 生成后缓存”的伪确定性方案。

### Q0-3 无状态实体与确定性生成的关系；支持玩家修改需要什么

> 为什么“无状态实体”和“确定性生成”互为充要？如果立方体允许被玩家修改，架构需要增加什么？

**解答：**

**两个方向的论证：**

- 无状态 ⇒ 必须确定性：既然 CPU 端不为任何立方体保存对象，区块被销毁时其信息**彻底消失**。当玩家走回来、区块需要重建时，唯一可用的输入只有区块坐标。若生成不是坐标的纯函数，重建出来的内容就会和上次不同——玩家会看到“世界变了”，无限世界的假象立刻穿帮。
- 确定性 ⇒ 无状态是安全的：既然任何时刻都能由坐标凭空重算出完整内容，那么保存实体状态就是纯冗余——状态可以安全地不存在。

**若允许玩家修改（如挖掉一个立方体）：**

纯函数模型被打破，因为世界内容不再只由坐标决定，还由“历史操作”决定。标准解法是**基线 + 稀疏差异层（diff/overlay）**：

- 基线：仍由确定性函数生成（零存储）。
- 差异层：一个 `Map<区块坐标, 修改记录>`，只记录被玩家改动过的位置（如“(cx,cy,cz) 区块内局部坐标 (1,2,3) 已删除”）。
- 重建区块 = `generateChunk(...)` 输出，再应用该区块的差异记录。

代价：内存不再是严格常数，而是随**修改量**（而非探索量）线性增长——这通常可接受，也正是 Minecraft 类游戏的实际架构。理解这条边界，就理解了本项目“不可修改”假设换来了什么。

### Q0-4 Float32 在 10⁵ 量级的精度

> `Float32` 在数值 100,000 附近的最小可分辨增量约是多少？这对相机平滑移动意味着什么？

**解答：**

Float32 有 23 位尾数。数值 100,000 落在区间 `[2¹⁶, 2¹⁷) = [65536, 131072)`，该区间内相邻可表示数的间距（ULP）为 `2^(16-23) = 2⁻⁷ ≈ 0.0078`；到 `[2¹⁷, 2¹⁸)` 区间则翻倍为 ≈ 0.0156。即在十万量级，坐标只能以约 **1/128 ~ 1/64 个世界单位** 为步进取值。

对相机平滑移动的影响：

1. **位置量化**：低速移动时每帧位移可能小于 1 ULP，位置“原地不动”，然后突然跳一格——表现为顿挫感。以 8 单位/秒、60 FPS 计，每帧 0.133 单位 ≈ 17 ULP，行走尚可接受；但慢速漂移、鼠标微调、缓动动画的末端会明显量化。
2. **灾难性消去**：视图变换中大量出现“两个相近大数相减”（如 `顶点世界坐标 − 摄像机坐标`，都是 10⁵ 量级），相减后的小差值只剩下少数有效位，误差被放大到肉眼可见——这就是顶点抖动（vertex jitter）的直接来源。
3. **着色器内动画劣化**：`sin(worldPos.x * freq + time)` 中 worldPos 本身已不精确，波形会出现台阶。

这就是“伪无限”的精度墙，以及附录 C「浮动原点」存在的理由：让参与计算的数值始终保持在原点附近的小量级。

### Q0-5 每帧 16.6 ms 的预算分配

> 给渲染数据流的每一步分配耗时预算，保证 60 FPS。

**解答（一种合理的分配，数值可论证即可）：**

| 环节 | 预算 | 依据 |
|---|---|---|
| 输入采集 + 摄像机更新 | ≤ 0.2 ms | 纯标量运算，几十次乘加 |
| 世界管理（区块进出） | 摊销后 ≤ 1 ms（跨界帧峰值 ≤ 2 ms） | 仅跨界时执行；配合生成队列摊销 |
| CPU 剔除 + 影子缓冲压缩写入 | ≤ 3 ms | 2.2 万实例的循环，每实例约几十条指令 |
| bufferSubData 上传 | ≤ 1 ms | ≈ 1.3 MB，正常应远快于此；超了就怀疑隐式同步 |
| draw 提交 + 其余 GL 调用 | ≤ 0.5 ms | 单 draw call，开销极小 |
| **CPU 合计** | **≤ 6 ms** | 给 GPU 渲染和浏览器合成留出余量 |

要点不在具体数字，而在两个意识：① 总预算 16.6 ms 中 CPU 侧只能占一部分，因为 GPU 渲染与合成同样消耗流水线时间；② 给每个环节定了预算，性能退化时才能用 Performance 面板**逐项对账**，而不是面对一个笼统的“变卡了”。

### Q0-6 每帧必做 vs 可摊销

> 哪些步骤是每帧必须执行的？哪些可以摊销？

**解答：**

**每帧必做**（依赖每帧都在变的输入——摄像机姿态与时间）：

- 输入采集、摄像机更新；
- CPU 剔除（距离剔除与视锥剔除的结果随摄像机每帧变化）；
- 影子缓冲压缩写入与上传、draw call。

**可摊销 / 按需触发：**

- **区块窗口更新**：只有摄像机跨越区块边界时活跃集合才变化，无需每帧对比集合。
- **区块内容生成**：跨界时新增的区块可放入队列，每帧限额生成 K 个（近者优先），把生成成本摊到多帧。
- **HUD/统计刷新**：每 10 帧更新一次足够。

判别标准：**该步骤的输出依赖的输入，变化频率是多少？** 输入每帧变 → 每帧算；输入低频变 → 事件触发；输入高频变但输出容忍延迟 → 摊销。

### Q0-7 CPU/GPU 边界穿越

> 数据在哪些环节穿越了 CPU/GPU 边界？为什么要压到最少？

**解答：**

本项目中的穿越点：

1. 每帧一次 `bufferSubData`（实例矩阵，≈ 1.3 MB，CPU → GPU）；
2. 每帧少量 uniform 更新（投影/视图矩阵、时间、光照参数，几百字节，CPU → GPU）；
3. draw call 本身（命令提交）。
4. （本项目刻意不存在的）GPU → CPU 回读，如 `readPixels`、`getError`、`clientWaitSync`。

要压到最少的原因：

- **每次 API 调用有固定驱动开销**（参数校验、命令编码），与数据量无关。100 次 1 KB 的上传远慢于 1 次 100 KB 的上传——这直接论证了“压缩写入 + 单次上传”的设计（Q3-2）。
- **隐式同步风险**：若 GPU 尚在读取某缓冲区，CPU 对它的写入会迫使驱动等待或暗中拷贝（停顿数毫秒）。
- **回读是最毒的**：任何 GPU → CPU 读取都要求把异步流水线排空（flush + 等待），一次就能毁掉整帧。CPU 与 GPU 本质是一条深流水线上的生产者与消费者，最佳状态是单向投喂、永不回头。

### Q0-8 WorldManager 为什么不持有 gl

> 为什么 `WorldManager` 不应该持有 `gl`？对单元测试和 Web Worker 化分别带来什么好处？

**解答：**

`WorldManager` 的本质职责是**计算**：输入摄像机区块坐标，输出 `Float32Array` 里的实例矩阵与计数。这个职责不需要 GPU 的任何能力，持有 `gl` 只会把“计算”与“提交”耦合。

具体收益：

1. **单元测试**：不持有 gl，模块可在 Node / Vitest 环境直接运行——无需 mock 整个 WebGL 接口（mock GL 是出了名的费力且测不到真问题）。可以直接断言“摄像机从 A 区块移到 B 区块后，活跃集合应增加哪些键、释放哪些槽位、缓冲区哪一段被写入了什么”。
2. **Web Worker 化**：Worker 中没有主线程的 canvas 上下文。如果 WorldManager 是纯计算模块，把它整体搬进 Worker 只需要解决消息传递（`postMessage` + `Transferable` 转移 `Float32Array` 所有权，零拷贝），主线程只剩“收到数组 → 上传 → 绘制”。若它持有 gl，则要先做一次伤筋动骨的解耦手术。
3. **附带收益**：上下文丢失恢复时（Q2-4），世界状态无需任何处理——它根本不知道 GL 出过事故。

## 阶段一：纯函数高性能数学库

### Q1-1 GC 实验的预期结果

> 「每帧 new 一万个 Float32Array」与「复用预分配数组」的 GC 轨迹分别什么样？为什么帧时间尖刺比平均帧时间更值得关注？

**解答：**

- **每帧分配版**：Memory 曲线呈**锯齿状**——堆占用持续爬升，到达新生代阈值后触发 minor GC 瞬间回落，循环往复。Performance 面板的主线程轨道上会出现周期性的 GC 块（哪怕每次只有 1~3 ms），与之对应的帧会超出 16.6 ms 预算。
- **预分配复用版**：内存曲线是一条**平线**，无 GC 事件，帧时间方差小。

为什么尖刺比平均值重要：人眼对**帧率波动**远比对绝对帧率敏感。稳定 40 FPS 的观感往往好于“平均 58 FPS 但每秒丢一帧”——丢帧表现为可感知的“跳一下”。GC 暂停恰好是典型的尖刺来源：它不降低平均值（摊到几百帧里微不足道），但每次发作都精确制造一次掉帧。因此性能指标要看 p95/p99 帧时间（阶段五），而消灭分配就是消灭尖刺的根源。

### Q1-2 cross(a, b, a) 别名问题

> `cross(a, b, a)`（out 与输入重叠）会发生什么？如何在不引入堆分配的前提下修复？

**解答：**

叉乘公式为：

```
out[0] = a[1]*b[2] − a[2]*b[1]
out[1] = a[2]*b[0] − a[0]*b[2]
out[2] = a[0]*b[1] − a[1]*b[0]
```

当 `out === a` 时，第一行写入 `out[0]` 即覆盖了 `a[0]`；而第二、三行的计算还要读取 `a[0]`——读到的已是污染后的值。结果是**静默的错误数据**：不报错、不 NaN，只是数值不对，且只在调用方恰好传了重叠参数时发作，极难排查。

修复（零分配）：函数内部先把全部输入分量读入**局部标量变量**，再写 out：

```
const ax = a[0], ay = a[1], az = a[2];
const bx = b[0], by = b[1], bz = b[2];
out[0] = ay*bz − az*by; ...
```

局部 number 变量由 JIT 放在寄存器/栈上，不产生堆分配。`mat4.multiply` 同理（16 个局部标量）。配套措施：为每个有风险的函数写一个“out 与输入同体”的单元测试，对照非别名调用的结果。

### Q1-3 暂存器策略与使用契约

> 每帧上万次向量运算，中间结果放哪里？模块级暂存向量的使用契约应包含哪些禁令？

**解答：**

两种可行方案：

1. **模块级暂存**（如 `mat4.ts` 顶层的 `_tmpVec0/_tmpVec1`）：实现最简单，但本质是隐藏的全局可变状态，必须附带严格契约：
   - **禁止跨 `await` / Promise 回调持有**——异步恢复后暂存器可能已被别人改写；
   - **禁止在递归或会重入的函数中使用**——内层调用会覆盖外层的中间值；
   - **禁止把暂存器引用返回给调用方或存入任何数据结构**——它的生命周期只有“当前同步代码段”；
   - **禁止跨线程**——每个 Worker 有自己的模块实例，天然隔离，但不能通过共享内存传递。
2. **调用方自带暂存**（更稳妥）：每个系统模块（Camera、WorldManager、Renderer）在构造时预分配自己专用的 scratch 数组。空间略有冗余，换来的是“谁分配、谁使用、互不可见”，重入与异步问题天然消失。

工程建议：数学库内部实现用局部标量（Q1-2 的方法，完全无共享状态）；跨多次调用的中间向量由调用方自带。模块级暂存只作为最后的便利手段，且命名加 `_tmp` 前缀明示危险。

### Q1-4 点乘与叉乘的几何语义

> `dot(forward, toTarget) > 0.999` 意味着什么角度？点乘符号、叉乘模长的几何语义及其在本项目中的用途？

**解答：**

- **点乘（单位向量）= 夹角余弦**。`cos θ > 0.999` ⇒ θ < arccos(0.999) ≈ **2.56°**，即目标几乎在视线正前方。可用于“准星吸附”“注视高亮”类功能。
- **点乘符号 = 前后半空间判定**：`dot(n, p − p₀) > 0` 表示点 p 在过 p₀、法线为 n 的平面正侧。这正是**视锥剔除的原子操作**（阶段四步骤 4）：六个平面各做一次符号判断。也是背面剔除的数学本质（面法线与视线方向的点乘符号）。
- **叉乘模长 = 两向量张成的平行四边形面积**（三角形面积 × 2）。面积趋近 0 即两边近乎平行——可检测**退化三角形**；同样的退化判断也出现在 lookAt 中 forward 与 worldUp 平行时（Q1-11）。
- **叉乘方向 = 构造正交基**：lookAt 的 r、u 就是两次叉乘搭出来的，欧拉角摄像机的 right 向量同理。

一句话总结：点乘回答“多对齐/哪一侧”，叉乘回答“垂直方向是什么/张成多大”。本项目里它们分别支撑了视锥剔除与摄像机正交基两大系统。

### Q1-5 平移分量的下标

> 平移分量 (tx, ty, tz) 在列主序数组中是哪三个下标？答错说明什么？

**解答：**

**m12, m13, m14**（第 4 列的前三个元素，`col*4 + row = 3*4 + 0/1/2`）。

若答成 m3, m7, m11，说明你在用**行主序**思维：把数学书上“平移在第四列”直觉地映射成了“每行最后一个元素”。这是 C/数学惯例（行主序）与 OpenGL 惯例（列主序）的经典冲突。后果链：往 m3/m7/m11 写平移 → 这三个位置实际是矩阵最底行的投影项 → 物体不平移反而产生诡异的透视扭曲或消失。`uniformMatrix4fv` 的 transpose 参数在 WebGL 中强制为 false，没有“传错了再转置回来”的退路，所以这张下标映射表必须刻进肌肉记忆。

### Q1-6 复合变换顺序

> 先绕自身 Y 轴旋转 β，再绕世界 X 轴旋转 α，最后平移到 (Tx, Ty, Tz)，连乘顺序？如何验证？

**解答：**

最终变换为 **`M = T · Rx(α) · Ry(β)`**，作用于点 p：`p' = T · Rx · Ry · p`。

推理规则：矩阵作用从**最靠近 p 的开始**（从右往左读）。“先做”的变换写在最右边。Ry 最先作用（此时物体还在自身原点，绕自身轴 = 绕世界轴，二者重合）；然后 Rx；最后 T 把结果搬到目标位置。如果把 T 放在最右边，旋转会绕世界原点把已平移的物体“抡”出去——这是最常见的复合顺序 bug 的视觉症状（物体绕着奇怪的远点公转而非自转）。

代码层面：`multiply(Rx, Ry, temp); multiply(T, temp, out);`

**验证方法**：取 p = (1, 0, 0)，β = 90°（绕 Y）。`Ry(90°)·p = (0, 0, −1)`（+X 转向 −Z，按标准右手系 Ry 定义）。再取 α = 90°（绕 X）：`Rx(90°)·(0,0,−1) = (0, 1, 0)`。最后加平移即 `(Tx, Ty+1, Tz)`。手算出这个预期值，写成单元测试断言。每个旋转函数至少配一个这样的“90° 转特殊点”测试——90° 的 sin/cos 是精确的 0 和 1，结果无舍入歧义。

### Q1-7 本项目是否需要逆转置法线矩阵

> CPU 端矩阵只含旋转 + 平移，法线矩阵需要逆转置吗？着色器内的顶点位移动画会破坏法线正确性吗？

**解答：**

**CPU 端：不需要。** 旋转 + 平移构成刚体变换，其 3×3 线性部分是正交矩阵，满足 `(M⁻¹)ᵀ = M`。直接用 `mat3(modelMatrix)` 变换法线即可，连归一化都理论上多余（正交变换保长），保留 normalize 只是防插值误差。

**着色器内动画逐项分析：**

- **整体平移型**（指南中的 wave：基于 instancePos 计算、对整个立方体加同一偏移）：刚体平移，法线**完全不受影响**。注意关键区别——wave 的输入是实例位置而非顶点位置，所以同一立方体的所有顶点位移相同。
- **均匀缩放型**（`a_position * scale`，scale 为标量）：法线方向不变、长度变化，**归一化后即正确**。
- **非均匀缩放**（如只沿 Y 拉伸）：法线方向真正改变，需要按 `normal / scale` 逐分量修正后归一化（等价于对角缩放阵的逆转置）。
- **逐顶点位移**（若 wave 改为基于顶点世界坐标，立方体被“剪切”变形）：表面几何变了，严格正确的法线需要对位移场求偏导重建。通常的工程选择是接受误差（位移幅度小、光照差异不可察觉），但**必须知道自己在近似**。

结论：在当前选型（实例级平移 + 均匀缩放）下，`mat3(model)` + normalize 严格正确，零额外开销。把这个前提写成注释，未来改动画类型时它就是警报器。

### Q1-8 m[11] = −1 的作用；near = 0 的后果

> `m[11] = -1` 的作用是什么？如果 near 设为 0 会发生什么？

**解答：**

**m[11] = −1 的作用**：在列主序约定下，m[11] 位于第 4 行第 3 列，它把视空间坐标的 `−z` 写入裁剪坐标的 **w 分量**（`w_clip = −z_view`）。随后硬件执行透视除法 `(x, y, z) / w`，于是“越远的点除以越大的数”——**透视收缩（近大远小）正是这一个 −1 制造的**。正交投影没有它（w 恒为 1），所以没有近大远小。同时它使 w 携带了深度信息，供透视校正插值（varying 除以 w 再插值）使用——没有它，贴在斜面上的纹理/颜色插值都会扭曲。

**near = 0 的后果**：代入公式 `m[14] = −2·far·near/(far−near)`，near = 0 使 m[14] = 0，深度映射退化为 `z_ndc = −m[10]` 的病态形式；本质问题是深度映射 `z_ndc ∝ 1/z` 在 z → 0 处发散——**全部深度精度被压缩到紧贴摄像机的无穷小邻域，其余整个场景共享几乎同一个深度值**，深度测试完全失效（满屏 Z-fighting）。这就是 near 永远不能为 0、且应在不裁穿场景的前提下**尽量取大**的原因：near 从 0.01 提高到 0.1，远处深度精度约改善 10 倍。

### Q1-9 Z-fighting 与 Reversed-Z

> 为何远处的 Z-fighting 更严重？反向 Z 为什么能缓解？WebGL 2.0 中实现需要改哪几处？

**解答：**

**为何远处更严重**：标准投影下 `z_ndc` 与 `1/z_view` 成线性关系，是一条双曲线——z_view 在 near 附近变化时 z_ndc 变化剧烈（精度充裕），在 far 附近 z_ndc 几乎不动。以 near=0.1、far=300 为例，超过一半的 NDC 深度区间被分给了摄像机前几米。远处两个面即使相隔几十厘米，量化到深度缓冲后可能是同一个值，深度测试结果随视角微变而翻转，表现为闪烁条纹。

**反向 Z 的原理**：把映射翻转为 near→1、far→0 后，配合**浮点**深度缓冲使用。IEEE 浮点的可表示数在 0 附近指数级密集——而翻转后“远处”恰好落在 0 附近。于是“浮点在 0 附近密”与“1/z 在远处疏”两条非线性**互相抵消**，深度精度在全距离范围内近似均匀。这是现代引擎（UE、Frostbite 等）的标配。

**WebGL 2.0 实现要点**（成套修改，缺一不可）：

1. 投影矩阵改为 near/far 翻转的形式（或左乘一个 z' = −z 的翻转）。
2. 深度测试函数 `gl.depthFunc(gl.LESS)` 改为 `gl.GEQUAL`。
3. `gl.clearDepth(1.0)` 改为 `clearDepth(0.0)`（“最远”现在是 0）。
4. `gl.depthRange` 保持 [0,1] 不动（WebGL 的 depthRange 会钳位，不能像桌面 GL 那样设 [1,0]，翻转必须做在投影矩阵里）。
5. 理想情况需要浮点深度附件（FBO + `DEPTH_COMPONENT32F`）才能吃满收益；默认帧缓冲的定点深度下收益减半但仍为正。

### Q1-10 视图矩阵平移为何是负点乘

> 为何视图矩阵的平移部分是 `-dot(r, eye)` 而不是简单的 `-eye`？

**解答：**

摄像机的世界变换是“先旋转到姿态 R，再平移到 eye”：`C = T(eye) · R`。视图矩阵是它的逆：

```
V = C⁻¹ = (T(eye) · R)⁻¹ = R⁻¹ · T(eye)⁻¹ = Rᵀ · T(−eye)
```

注意求逆后**顺序反转**：先反平移、后反旋转。把 `Rᵀ · T(−eye)` 乘开，平移列变成 `Rᵀ · (−eye)`——即 `−eye` 这个向量**被旋转矩阵的转置作用过**。Rᵀ 的三行恰是基向量 r、u、−f，因此该列的三个分量就是 `−dot(r, eye)`、`−dot(u, eye)`、`dot(f, eye)`。

直观理解：视图变换回答的问题是“这个世界点**在摄像机自己的坐标系里**坐标是多少”。平移量自然也要表达在摄像机的坐标轴上——`dot(r, eye)` 正是“eye 在 right 轴上的投影长度”。如果直接用 `−eye`（世界轴下的分量）填进去，等于把一个世界系向量塞进了摄像机系的格子里，旋转任意非零角度后必错。验证方法：构造 V 后计算 `V · (eye, 1)`，结果必须是 `(0, 0, 0, 1)`——摄像机在自己的坐标系里位于原点。把这条写进单元测试。

### Q1-11 lookAt 的退化情形

> 当 f 与 worldUp 平行（垂直俯视）时，构造过程在哪一步退化？产生什么后果？与阶段四的哪个设计相关？

**解答：**

退化点在第二步 `r = normalize(cross(f, worldUp))`：

1. f 与 worldUp 平行（夹角 0° 或 180°）时，`cross(f, worldUp)` 的模长 = |f||up|·sin0 = **0**，得到零向量；
2. `normalize(零向量)` 做 0/0 除法 → 三个分量全为 **NaN**；
3. NaN 沿着 `u = cross(r, f)` → 视图矩阵 → 顶点着色器的 MVP 乘法逐级传播，所有顶点的裁剪坐标变为 NaN；
4. GPU 对 NaN 坐标的图元直接丢弃——**整屏内容消失**，且控制台一个错误都没有（NaN 不是 GL 错误）。

数值上更隐蔽的是“接近平行”：夹角 0.1° 时叉乘模长极小，归一化虽不产生 NaN，但有效位数所剩无几，right 向量方向剧烈抖动，表现为俯视时画面疯狂旋转抖动。

这正是**阶段四 pitch 钳位**（`pitch ∈ [−π/2+ε, π/2−ε]`）的数学根源：钳位不是“手感偏好”，而是在输入端保证 f 与 worldUp 的夹角永远有一个安全余量，使 lookAt 的正交基构造远离退化区。防御深度上还可以在 `normalize` 内加零向量保护（返回零向量或备用轴），但那是最后防线，第一防线永远是约束输入域。

## 阶段二：WebGL 2.0 基础管线与显式封装

### Q2-1 上下文选项的选取

> `antialias`、`alpha`、`powerPreference` 各应取什么值？`alpha: false` 为什么通常能提升合成性能？

**解答：**

- **`antialias: true`** 起步阶段建议开启（立方体世界全是硬边，锯齿很显眼）。它启用的是默认帧缓冲的 MSAA，成本是更大的渲染目标带宽。后期若片元成为瓶颈，可改为 false 并自行权衡（或上后处理 AA）。注意它只是“请求”，驱动可拒绝。
- **`alpha: false`**：声明画布**不透明**。为 true（默认）时，浏览器合成器必须把 canvas 当作带 alpha 的图层与页面背景做混合——每帧多一次全屏混合，且某些平台为此放弃 overlay 优化路径。false 则合成器可以整块不透明拷贝/直接扫描输出。本项目画布铺满全屏且不需要透出页面背景，false 是纯赚。
- **`powerPreference: 'high-performance'`**：在双显卡设备（核显 + 独显笔记本）上请求独显。默认值可能为了省电选核显，帧率差距可达数倍。代价是耗电，对一个实时 3D 应用这是合理交换。
- 顺带明确其余项：`depth: true`（默认，必须）、`stencil: false`（不用模板）、`preserveDrawingBuffer: false`（默认；true 会禁止交换链丢弃旧帧，明显伤性能，只有截图需求才临时开）。

### Q2-2 背面剔除的收益与“破洞”诊断

> 背面剔除能省掉立方体约几成的片元工作？开启后立方体「破了几个洞」说明什么？

**解答：**

**收益**：凸体（立方体）的任意视角下，恰好一半的面背向观察者（6 面中最多见 3 面）。背面剔除在**图元装配阶段**（光栅化之前）就丢弃这些三角形，它们的片元着色器一次都不会执行——直接砍掉约 **50%** 的潜在片元工作。注意这与深度测试不同：深度测试发生在更晚的阶段，被遮挡片元可能仍产生开销（取决于 Early-Z 是否生效）；背面剔除是更早、更便宜的一刀。在 2 万实例的场景里这是免费的 2 倍片元余量。

**破洞诊断**：剔除的判据是三角形在屏幕上的**绕序**（默认 CCW = 正面）。立方体某几个面消失（从洞里能看到内部背面），说明这几个面的索引顺序写反了——从外部看是顺时针，被当作背面剔除。这其实是**免费的几何数据校验器**：与其关掉 CULL_FACE 掩盖问题，应该修正索引绕序。检验方法：盯着每个面的三个索引，按右手定则确认从立方体外侧看去为逆时针。所有面修对后，开启剔除的立方体在任何角度都应完整。

### Q2-3 u_time 为什么不能用绝对时间戳

> 为什么 `u_time` 不能直接用绝对时间戳？长时间运行后动画为什么会“变糙”？对策？

**解答：**

问题出在 **Float32 的相对精度**。`performance.now()` 返回毫秒，运行 3 小时 ≈ 1.08 × 10⁷ ms。该量级下 Float32 的 ULP = 2^(23−23) = **1 ms 以上**——也就是说，把这个时间戳从 JS 的 double 转成着色器的 float 时，连毫秒级变化都开始无法表示。后果链：

1. `u_time` 在相邻帧间量化到同一个值或大步跳变 → 动画出现台阶感、不再平滑；
2. `sin(worldPos.x * freq + u_time)` 中，大参数进入 sin 前的区间规约（argument reduction）在 fp32 下丢失精度 → 波形畸变、相邻立方体相位关系错乱；
3. 劣化是**渐进**的：开发时测十分钟一切正常，挂机一晚后动画“变糙”——典型的逃过测试的 bug。

**对策（成套）：**

1. **累积制**：`time += min(deltaTime, 0.1)`，以秒为单位累积。这同时解决了“切后台回来 deltaTime 巨大导致动画瞬跳”的问题（钳位）。
2. **周期回绕**：动画全部由 sin/cos 等周期函数构成时，把累积时间对总周期（如 `2π × 1000 / freq` 的公倍数，或简单地对一个大周期如 3600 秒）取模，保证传入着色器的数值永远在小量级。回绕瞬间因相位连续而无任何视觉跳变。
3. 单位用**秒**而非毫秒（数值小 1000 倍，精度余量多 ~10 位）。

### Q2-4 上下文恢复后重建什么

> 上下文恢复后哪些对象需要重建？哪些不需要？与阶段零哪条约束相互印证？

**解答：**

**必须重建——一切 GPU 侧资源**（上下文丢失即显卡侧状态全部蒸发）：

- 所有 `WebGLProgram` / `WebGLShader`（重新编译链接）；
- 所有 VBO / IBO / VAO（重新分配并把 CPU 侧影子数据重新上传）;
- 所有全局状态设置（`enable(DEPTH_TEST)`、clearColor 等——它们属于上下文，随之重置）；
- Uniform location 缓存（program 是新的，location 全部作废）。

**无需重建——一切纯 CPU 数据**：

- 数学库（无状态，谈不上重建）；
- `WorldManager` 的活跃区块 Map、槽位分配器、影子 `Float32Array`（数据原封不动，恢复后直接整体重新上传一次即可）；
- Camera 状态、输入状态、累积时间。

这与阶段零 **“world 不依赖 GL / 依赖方向单向”** 的约束相互印证：因为世界状态与 GPU 资源在架构上是两个世界，恢复流程退化为机械的“重建 core/render 层 + 重传数据”，不需要任何业务逻辑参与。若 WorldManager 里散落着 `gl.bufferSubData` 调用，恢复逻辑就得侵入世界管理代码。实践提示：把所有 GPU 资源创建收口到一个 `initGLResources()` 函数，上下文恢复 = 再调一次它——能否做到“一个函数恢复”，本身就是资源管理是否收口的试金石。

### Q2-5 getUniformLocation 的查询成本与“拿不到 location”

> `gl.getUniformLocation` 为什么不该在渲染循环中调用？对不存在的 uniform 返回什么？“声明了却拿不到”的头号原因？

**解答：**

- **为什么不该每帧调**：它是带字符串参数的同步驱动查询——内部要做字符串比对/哈希查找，且作为 GL 调用本身有固定开销。每帧对几十个 uniform 各查一次，纯属把初始化成本搬进热路径。正确做法：链接成功后用 `getProgramParameter(ACTIVE_UNIFORMS)` + `getActiveUniform` 枚举所有活跃 uniform，一次性建好 `Map<name, location>`。
- **不存在时返回 `null`**。随后 `gl.uniformXXX(null, ...)` 会被**静默忽略**（不报错）——这是双刃剑：程序不崩，但你设置的值悄无声息地丢了。建议在封装的 setUniform 里对 null location 做**一次性 console.warn**（warn 一次后记住，避免每帧刷屏）。
- **“声明了却拿不到”的头号原因**：GLSL 编译器的**死代码消除**。一个 uniform 若未对最终输出产生贡献（比如你声明了 `u_lightColor` 但调试时临时把输出改成了纯法线可视化），编译器直接把它裁掉，它就不是 active uniform，查询返回 null。症状：昨天还好好的 uniform 今天拿不到了——往往是因为你注释掉了使用它的那行代码。理解这一点，就不会再浪费时间怀疑拼写。

### Q2-6 插值后法线为何要重新归一化

> 法线在顶点着色器已归一化，为什么插值到片元后必须重新归一化？

**解答：**

光栅化对 varying 做的是**线性（重心坐标）插值**，而线性插值**不保长度**。几何直观：两个单位向量端点连一条直线段，这条弦上的每个点都在单位球面**内部**——除两端点外，插值结果的长度全部 < 1。两法线夹角越大，中间收缩越狠（夹角 90° 时中点长度仅 ≈ 0.707）。

不归一化的后果：`dot(N, L)` 的值被整体缩小，面片**中部偏暗、顶点处偏亮**，出现以三角形为单位的暗斑纹理；镜面项经过 `pow(x, shininess)` 放大后畸变更明显。

所以片元着色器第一行 `N = normalize(v_normal)` 不是防御性冗余，而是**数学必需**。本项目立方体的每个面内法线恒定（4 个顶点同法线，插值结果不变），影响暂不可见——但这恰是危险所在：换成球体等平滑着色几何时问题立刻显形。从一开始就写对，不给未来埋雷。

### Q2-7 漫反射为何必须 clamp 到 0

> 为什么必须 `max(dot(N, L), 0.0)`？背光面如果出现负光会怎样？

**解答：**

`dot(N, L)` 在表面背向光源时为**负值**（夹角 > 90°）。物理上这意味着“光照不到”，正确贡献是 0；数学上若不 clamp，负值会进入颜色合成：

1. **负漫反射**会从环境光中**倒扣亮度**：`ambient + (负数)·lightColor` 使背光面比纯环境光照还暗，甚至被压到负颜色（最终被钳到黑，表现为背光面死黑一片且与环境光设定矛盾）。
2. 镜面项后果更怪诞：`pow(负数, shininess)` 当 shininess 非整数时在 GLSL 中是 **undefined / NaN**（pow 对负底数未定义），NaN 进入颜色输出在不同 GPU 上表现不一（黑块、白块、闪烁）——一个跨设备不可复现的 bug 就此诞生。所以镜面项的 `max(dot(N, H), 0.0)` 同样不可省。

物理含义：clamp 实现的是**单面受光模型**——辐照度在入射角超过 90° 后为零，而非变负。半程向量的 dot 同理。规则可推广为：**任何将进入 pow、sqrt、除法的光照中间量，先想清楚其合法域，并在边界处钳位**。

### Q2-8 立方体为什么需要 24 个顶点

> 为什么立方体不能只用 8 个顶点？只用 8 个会得到什么视觉效果？

**解答：**

顶点在 GPU 中是“**属性元组**”而非“空间位置”：位置 + 法线（+ UV...）整体构成一个顶点。立方体每个角点被 3 个面共享，而这 3 个面的**法线互相垂直**——同一位置需要 3 套不同属性，就必须是 3 个不同的顶点。8 角 × 3 = **24 顶点**，每面 4 顶点 + 2 三角形 = 36 索引。

**只用 8 个顶点的效果**：每个角点只能存一条法线，通常取三面法线的平均（指向角点对角线方向，如 normalize(1,1,1)）。插值后整个立方体表面的法线连续过渡——光照计算把它当成**平滑曲面**，渲染结果是一个“充了气的圆鼓鼓立方体”观感：棱角处明暗渐变、没有面与面的清晰分界，立方体特有的硬朗块面感完全丢失。

通用规律：**顶点是否复制，取决于属性是否连续**。平滑物体（球）共享顶点（法线本来连续）；硬边物体（立方体）按面复制；同理，UV 接缝、材质分界处也需要复制顶点。“硬边 = 属性不连续 = 顶点分裂”这个等式在一切建模/几何生成中通用。

### Q2-9 解绑 VAO 后调用 vertexAttribPointer 污染谁

> 解绑 VAO 后再调用 `vertexAttribPointer` 会污染谁？由此推出属性配置必须在什么时机完成？

**解答：**

WebGL2 中 `bindVertexArray(null)` 并非“无 VAO”，而是切回**默认 VAO**（上下文自带的那个）。此后所有 `vertexAttribPointer` / `enableVertexAttribArray` / ELEMENT_ARRAY_BUFFER 绑定都会写进**默认 VAO 的状态**：

1. 你以为在配置自己的 VAO，实际配置“丢”进了默认 VAO——你的 VAO 缺属性，绘制时报 attribute 未启用或画面全无；
2. 默认 VAO 被悄悄污染，若其他代码路径依赖默认 VAO 的干净状态，产生超距离的诡异 bug；
3. 若此时 ARRAY_BUFFER 也没绑定，`vertexAttribPointer` 直接抛 `INVALID_OPERATION`（这是幸运的情况——至少报错了）。

**推论——属性配置的正确时序公式：**

```
bindVertexArray(vao)            // 1. 先绑 VAO（开始录制）
bindBuffer(ARRAY_BUFFER, vbo)   // 2. 绑数据源
vertexAttribPointer / enable    // 3. 配置被记录进 VAO（同时记下当前 VBO 的关联）
bindBuffer(ELEMENT_ARRAY_BUFFER, ibo)  // 4. IBO 绑定本身被 VAO 记录
bindVertexArray(null)           // 5. 解绑（停止录制）
```

两条易错细节：① VAO 记录的是“属性 ↔ VBO 的关联”，但**不记录** ARRAY_BUFFER 绑定点本身——绑 VBO 必须发生在调用 vertexAttribPointer 之前，而非“绑过就行”；② ELEMENT_ARRAY_BUFFER 则**属于** VAO 状态，解绑 VAO 之前不要去解绑 IBO，否则关联一起被解掉。

## 阶段三：高性能实例化渲染与无限世界引擎

### Q3-1 缓冲用途提示与每帧上传量

> `STATIC/DYNAMIC/STREAM_DRAW` 对驱动是什么性质的信息？本项目每帧上传字节量是多少？会成为瓶颈吗？

**解答：**

**用途提示的本质**：它们是给驱动的**性能提示而非功能约束**——无论指定哪个，读写行为都合法、结果都正确。驱动据此决定缓冲的物理存放位置与更新策略：`STATIC_DRAW`（写一次用多次）倾向纯显存；`STREAM_DRAW`（高频改写）倾向 CPU 可高效写入的内存（写合并内存/统一内存），并可能为其准备多缓冲机制以吸收写入。提示选错不会坏，只会慢——比如把每帧改写的缓冲标成 STATIC，某些驱动每次更新都走“搬回-修改-搬回显存”的慢路径。

**本项目的量**：上限 21,952 实例 × 64 B ≈ **1.34 MB/帧**；剔除后实际通常更小。60 FPS 下 ≈ **80 MB/s**。对比参考：PCIe 3.0 x16 理论 ~16 GB/s，集成显卡的统一内存更不存在拷贝——带宽占用约 0.5%，**不会成为瓶颈**。

真正可能出事的不是带宽而是**同步**：若驱动在 GPU 仍引用该缓冲时收到 bufferSubData，可能阻塞等待（一次 stall 数毫秒，远贵于传输本身）。所以排查顺序是：Performance 面板看到 bufferSubData 耗时异常 → 先怀疑隐式同步（试孤儿化/双缓冲）→ 而不是去“压缩数据量”。

### Q3-2 压缩写入 + 单次上传 vs 按区块多次上传

> 为什么“压缩写入 + 单次上传”优于“按区块多次 bufferSubData”？与槽位设计如何协调？

**解答：**

**为什么单次上传赢**（343 个区块 = 343 次 bufferSubData 的成本拆解）：

1. **每次调用的固定开销**：JS→浏览器进程→GPU 进程的命令通道，每个 GL 调用都有参数校验与编码成本。343 次小上传的固定开销总和远超 1 次大上传。
2. **同步检查次数**：每次 bufferSubData 驱动都要判断目标区间是否被在途绘制引用，343 次就是 343 次判断与潜在 stall 点。
3. **绘制连续性**：剔除后各区块的存活实例数不同，若按槽位原地上传，缓冲区中存活实例之间有“空洞”，无法用一个 `drawElementsInstanced(count)` 连续绘制——要么按区块发多个 draw call（违背单 draw 设计），要么 GPU 端处理空洞。压缩写入天然产出**致密的 [0, count) 区间**，一个 draw call 收尾。

**与槽位设计的协调——两层缓冲各司其职：**

- **影子缓冲（按槽位组织，跨帧持久）**：每个活跃区块占固定区间，区块生成时写入一次，之后不动。槽位分配器管理的是这一层。它是“数据库”。
- **上传缓冲（每帧重建，致密）**：每帧遍历活跃区块的槽位区间，对每个实例做距离/视锥剔除，存活者**追加拷贝**到上传缓冲前段，最后 `bufferSubData(0, count×64)` + draw(count)。它是“当帧视图”。

这样区块生成成本（哈希、拼矩阵）只在区块进入窗口时发生一次，每帧成本只剩“过滤 + 拷贝”，且上传与绘制都是单次连续操作。

### Q3-3 切比雪夫窗口 vs 球形窗口

> 为什么用切比雪夫距离（立方体窗口）而非欧氏距离（球形窗口）？改用球形能省多少区块？

**解答：**

**选切比雪夫的理由：**

1. **集合大小恒定且形状规整**：窗口永远是 (2R+1)³ 的立方体，槽位总数是编译期常数，free list 容量、VBO 容量都可静态确定。球形窗口在离散格点上的区块数虽然也固定，但形状不规整，遍历与推理都麻烦。
2. **成员判定是三次绝对值比较**：`max(|dx|,|dy|,|dz|) ≤ R`，无乘法；欧氏需要 `dx²+dy²+dz² ≤ R²`（也不贵，但更复杂）。
3. **遍历生成 idealSet 就是三重 for 循环**，与判定条件天然同构。
4. 工程顺序正确性：先用最简单的形状把滑动窗口算法做对，再优化形状——形状优化不触动算法骨架。

**球形能省多少**：立方体内切球体积比 = `(4/3)π(R+½)³ / (2R+1)³ ≈ π/6 ≈ 52.4%`，即**省约 47.6%** 的区块——被砍掉的全是立方体的 8 个角区域，而那些区块离摄像机最远（角距离是面距离的 √3 倍）、又在雾效中几乎不可见，性价比极低。实现上只需在三重循环内加一条欧氏距离判断；唯一要重新核算的是槽位总数上限（取球形区块数的精确计数而非 (2R+1)³）。作为附录 C 的第一项进阶，它是“一行判断换一半负载”的典型优化。

### Q3-4 为什么先回收后分配

> 为什么必须“先回收后分配”？用摄像机沿对角线跨区块的例子论证。

**解答：**

槽位总数 = 窗口大小 = (2R+1)³ = 343（R=3），即 free list 的资源恰好“量入为出”，**没有任何冗余**。

**对角线例子**：摄像机一步从区块 (0,0,0) 移到 (1,1,1)（对角跨界，三轴各 +1）。新旧两个 7×7×7 窗口的交集是 6×6×6 = 216 个区块，因此：

- 离开窗口的区块：343 − 216 = **127 个**（待回收）；
- 新进入窗口的区块：同样 **127 个**（待分配）。

**若先分配后回收**：分配 127 个新区块时，旧的 127 个还占着槽位——瞬时需求 = 216（保留）+ 127（旧）+ 127（新）= **470 个槽位 > 343 总量**。free list 在分配到第 (343−343)=0 个剩余后耗尽，程序要么崩溃要么被迫动态扩容（违背零分配承诺，且扩容后的容量再也缩不回去）。

**先回收后分配**：第一遍循环先把 127 个离开者的槽位压回 free list（此时 free list 有 127 个空闲），第二遍循环再为 127 个新区块弹出槽位——**峰值占用严格 ≤ 343**，恰好自洽。

更深的原理：这是资源池容量规划的通用定理——**资源池容量只需覆盖“稳态占用”的充要条件，是任何换入换出操作都先释放后获取**；只要有一处先获取后释放，容量就必须按“稳态 + 最大瞬时交换量”规划。一行代码顺序，决定了 37% 的容量冗余是否必要。

### Q3-5 一帧跨多个区块时算法是否正确

> 摄像机一帧内跨越多个区块（高速移动或大 deltaTime）时，该算法还正确吗？为什么？

**解答：**

**正确。** 关键在于算法是**声明式**的：每次 update 都从当前摄像机区块**从零重建**理想集合 `idealSet`，然后做两个集合的差运算（`active − ideal` = 回收，`ideal − active` = 分配）。整个推导中没有任何一步使用了“摄像机相对上一帧只移动了一格”的假设——哪怕摄像机瞬移 1000 个区块（极端：两个窗口完全不相交），算法的行为依然良定义：回收全部 343 个旧区块、分配全部 343 个新区块，结果正确（只是该帧生成压力大，由摊销队列吸收，见 Q3-7）。

**对比反面方案**就明白这个性质多宝贵：增量式算法——“摄像机向 +X 移了一格，则卸载 x = camCx−R−1 的那一平面、加载 x = camCx+R 的那一平面”。它每步只处理 49 个区块、看似更高效，但隐含“单步单格”假设：一帧跨两格、对角移动、传送、暂停后恢复，每一种都需要专门补丁，漏一种就出现“窗口与摄像机错位”的累积性 bug，且这种 bug 只在特定运动模式下复现。

这是“**声明式（描述目标状态）vs 增量式（描述状态变化）**”的经典权衡：声明式以少量常数开销（O(343) 的重建，见 Q3-6）买来对任意输入的无条件正确。React 的 reconciliation、Kubernetes 的期望状态调和是同一思想。指南要求把这一点写进注释，因为后人看到“每次全量重建”很容易手痒改成增量——注释要说清这是有意为之。

### Q3-6 idealSet 重建的开销量级

> idealSet 每次重建是 O((2R+1)³) ≈ 343 次循环，会是瓶颈吗？若 R 增大到 8 呢？

**解答：**

**R=3 时**：343 次迭代，每次做的事是拼一个字符串键 + 两次 Map 查询量级的操作。现代 JS 引擎单核每毫秒可执行数百万次简单操作，343 次 ≈ **微秒级**，且仅在跨界帧发生（玩家以 8 单位/秒移动、CHUNK_SIZE=4，约每 0.5 秒才跨一次界）。相对 16.6 ms 预算占比 < 0.1%，**绝不是瓶颈**。能给出这个量级估算（操作数 × 单操作成本 vs 预算）比记住结论重要——这是判断“要不要优化”的通用算法。

**R=8 时**：(2×8+1)³ = 17³ = **4,913** 个区块，迭代数增至 14 倍，本身仍只有几十微秒。但要警惕的瓶颈转移到别处：

1. **字符串键的分配**：每次重建拼 4,913 个 `"cx,cy,cz"` 字符串，违背零分配原则且制造 GC 压力。优化：整数打包键——把三个坐标各偏移后压进一个 number（如 `(cx+2¹⁹)·2⁴⁰ + (cy+2¹⁹)·2²⁰ + (cz+2¹⁹)`，53 位安全整数内），Map 换成以整数为键，分配归零。
2. **真正的大头**：实例总数 4,913 × 64 ≈ **31 万**，是 R=3 的 14 倍——每帧剔除循环、顶点处理、生成成本全部 ×14。R 的代价是三次方增长，窗口管理本身反而是最不需要担心的部分。

结论：idealSet 重建在任何合理 R 下都不是瓶颈；R 增大时先优化键的表示（消分配），并接受“实例数三次方爆炸”才是 R 的真实定价。

### Q3-7 跨界生成尖刺与摊销策略

> 跨界瞬间最多新增多少个区块？一次性生成会造成单帧尖刺吗？如何设计摊销并验证？

**解答：**

**新增量**：沿单轴跨一格，新增一个 7×7 的平面 = **49 个区块**（R=3）；对角跨界最多 127 个（Q3-4）。以 49 计：49 × 64 = **3,136 个实例矩阵**要在该帧内完成“哈希判定 + 三角函数 + 16 浮点数拼装写入”，按每矩阵 ~200 ns 估算约 0.6 ms——单独看可承受，但它叠加在正常帧负载之上，且 CHUNK_SIZE 或 R 稍大就线性/平方恶化（CHUNK_SIZE=8 时同样的跨界变成 2.5 万矩阵 ≈ 5 ms，必出尖刺）。帧时间曲线上的表现：平稳的 16 ms 序列中，每逢跨界出现一根孤立的尖峰——**周期性掉帧，行走时规律性顿挫**，恰好是体验上最敏感的模式。

**摊销设计：**

1. 跨界时不立即生成，把新区块坐标推入**待生成队列**；
2. 队列按到摄像机的距离排序（近者先生成——远处区块晚一两帧出现，被雾盖住无感）；
3. 每帧从队列头取最多 **K 个**区块执行生成（K 按预算定：单区块生成 ~0.1 ms，预算 1 ms 则 K=8~10）；
4. 未生成完的区块槽位暂不参与绘制（计数为 0），生成完毕才进入剔除/上传流程；
5. 细节：若某区块还在队列中就已再次离开窗口（玩家折返），从队列中移除，避免做废功。

**验证方法**：Performance 面板录制“持续直线移动 30 秒”，对比摊销前后的帧时间序列——摊销前每 0.5 秒一根尖刺（高度随 CHUNK_SIZE 增长），摊销后曲线平坦、尖刺消失，代价仅是远端区块延迟 1~6 帧出现（视觉上被雾完全掩盖）。同时在 HUD 上显示队列长度，确认其在持续移动下收敛而非无限堆积（K × 60 ≥ 每秒平均新增区块数即收敛）。

### Q3-8 整数坐标哈希的写法与质量要求

> 如何写一个「整数坐标 → [0,1) 伪随机数」的哈希且无分配？质量核心要求是什么？

**解答：**

**形态**：纯位运算/整数乘法的标量函数，输入三个整数与种子，输出 [0,1) 浮点，全程只用 number 局部变量——**零分配**天然满足。经典构造模式（Wang hash / PCG 输出函数风格）：

```
hash3(x, y, z, seed):
  h = (x * 73856093) ^ (y * 19349663) ^ (z * 83492791) ^ seed   // 大质数混合三轴
  h = (h ^ (h >>> 16)) * 0x45d9f3b                              // 雪崩轮 1：异或折叠 + 乘扩散
  h = (h ^ (h >>> 16)) * 0x45d9f3b                              // 雪崩轮 2
  h = h ^ (h >>> 16)
  return (h >>> 0) / 4294967296                                  // 无符号化后映射 [0,1)
```

（JS 中用 `Math.imul` 做 32 位乘法防溢出；`>>> 0` 转无符号。同一坐标要多个独立随机量——是否放置、相位、色相——用不同 seed 各调一次。）

**质量核心要求是“雪崩性”（avalanche）**：输入翻转任意 1 个比特，输出每个比特都应有 ~50% 概率翻转。本场景的输入是**高度规律的相邻整数坐标**（(5,0,7) 与 (6,0,7)），弱哈希（比如只做一次乘加）在这种输入下输出强相关——视觉直接暴露为**条纹、棋盘格、斜向纹理**等周期性图案，世界“看起来有人工规律”。乘大奇数（扩散到高位）与右移异或（折叠回低位）交替进行正是为了打碎这种相关性。

**验证**：① 直方图——大量样本均值 ≈ 0.5、分布均匀；② **视觉测试最有效**——把 hash 值直接渲染成一个平面的灰度图，人眼对条纹/格子图案的敏感度远超统计量，看一眼便知雪崩是否充分；③ 相邻性抽查——`hash(x,y,z)` 与 `hash(x+1,y,z)` 的序列做相关性检查。

### Q3-9 确定性的两个验证测试

> 如何验证生成的确定性？设计两个测试。

**解答：**

**测试一（单元级，字节等价）**：

```
对任意固定坐标 (cx, cy, cz)：
  buf1 = new Float32Array(N); generateChunk(cx, cy, cz, seed, buf1)
  buf2 = new Float32Array(N); generateChunk(cx, cy, cz, seed, buf2)
  断言 buf1 与 buf2 逐元素严格相等（含写入数量 count 相等）
```

强化版：两次调用之间穿插生成几十个**其他**区块再回来生成同一个——专门捕捉“生成器内部残留可变状态”（如复用的计数器、未重置的暂存数组、被上次调用推进过的伪随机序列）这类最常见的确定性破坏源。再加一组**跨进程**快照对比（把某固定坐标的输出序列化存为测试夹具，每次 CI 比对），排除“依赖 Math.random 后缓存”的伪确定性。坐标用例应覆盖正、负、零与大值（±10⁶）——负坐标哈希与 floor 行为是高发翻车点。

**测试二（集成级，往返一致）**：

```
1. 摄像机置于原点，渲染一帧，截取 canvas 像素（或记录活跃区块的实例计数指纹）
2. 程序化移动摄像机离开 ≥ 2R+1 个区块（确保原区块全部销毁、槽位被复用）
3. 原路返回精确坐标，再渲染一帧
4. 断言两帧像素一致（动画项需固定 u_time 后比较；或退一步比对各区块实例数/矩阵校验和）
```

测试二的价值在于覆盖**全链路**：哈希 → 生成 → 槽位分配（返回时拿到的很可能是不同槽位）→ 写入 → 渲染。它能抓住单元测试抓不到的问题，例如生成逻辑不小心依赖了槽位索引或 `gl_InstanceID`（Q3-11 的病根）。两个测试一微观一宏观，合起来才构成“销毁后可凭空重建”这一架构根基的回归防线。

### Q3-10 从均匀噪声到“地形感”

> 若想要“地形感”而非均匀噪声（如下方实心、上方稀疏），在哈希之上叠加什么？

**解答：**

均匀哈希的问题是**白噪声没有空间结构**——每个位置独立掷骰子，结果是均匀的“雪花”。地形感 = 放置概率/形态随空间**低频变化**。按复杂度递进的三层方案：

1. **高度门限（最简单，立竿见影）**：把放置判定从 `hash < 常数` 改为 `hash < density(y)`，density 随世界 y 坐标递减，如 `density(y) = clamp(1 − y / H_max, 0, 1)` 或更陡的 `smoothstep(H_top, H_bottom, y)`。效果：下方近实心、向上指数稀疏，出现“地壳 + 散块”的分层感。成本为零（一次比较），但水平方向仍是均匀噪声，没有山谷起伏。
2. **值噪声（Value Noise）做高度场**：在水平面上取稀疏格点（如每 8 单位一个），格点值用整数哈希生成，格点之间用 smoothstep 插值，得到连续的 `height(x, z)` 起伏面；放置条件改为 `y < height(x, z)`。这就有了**连绵的丘陵**。再叠加 2~3 个倍频（octaves，频率倍增、振幅减半）可得到大起伏套小细节的分形地形。关键约束：插值所用的格点哈希仍是坐标纯函数 → **确定性自动保持**，且区块可独立生成（计算 height 只需邻近格点，无跨区块依赖）。
3. **梯度噪声（Perlin/Simplex）**：格点存随机梯度向量而非标量，插值质量更高（无值噪声的方格状伪影）。原理同上，属于锦上添花。

共同的架构要点：无论叠加多复杂的噪声，**输入永远只有坐标与种子**——“地形感”是在纯函数框架内做出来的，Q0-2 的签名约束一寸不让。

### Q3-11 动画相位为何不能用 gl_InstanceID

> 波浪相位为什么必须用世界坐标而不是 `gl_InstanceID`？用后者会出现哪两类具体症状？

**解答：**

`gl_InstanceID` 是“**本次 draw call 中的第几个实例**”——它编号的是**缓冲区位置**，不是**空间位置**。而本架构中实例在缓冲区中的排列由两个与空间无关的因素决定：区块拿到哪个槽位（取决于 free list 的弹出顺序，即历史路径）、以及每帧剔除压缩后的紧致排列（取决于摄像机当帧姿态）。

**症状一：动画跳变（时间维度不连续）**。玩家走远再回来，同一个区块大概率分到**不同槽位**、其实例获得不同的 InstanceID → `sin(ID + time)` 的相位突变 → 肉眼可见“整片立方体的波浪动画瞬间换了一套节奏”。更糟的是用了每帧压缩方案后，**摄像机转个头**（剔除集合变化导致压缩后排列移位）都会让所有实例的 ID 漂移——画面每帧都在抽搐。这是 Q0-3“确定性”原则在动画维度的破产。

**症状二：波形不连贯（空间维度不连续）**。缓冲区中相邻的实例在空间中毫不相邻（不同区块、不同剔除结果交错排列），`sin(ID·freq + time)` 在空间中表现为**随机闪烁的噪声**，而非设计意图中“沿 X/Z 传播的连续波浪”——波动方程的自变量必须是空间坐标，波才成其为波。

**正确做法**：相位取自 `i_col3.xyz`（模型矩阵第 4 列 = 实例世界平移），它随实例数据本身走，与槽位、ID、剔除历史全部无关——同一个立方体无论何时何地被重建，动画相位严格复现，且相邻立方体相位连续。可总结为一条架构铁律：**渲染管线的瞬时编号（InstanceID、槽位、缓冲偏移）只能用于寻址，一切“内容语义”必须锚定在世界坐标上**。

### Q3-12 warp 内分支发散的代价；什么样的 if 无害

> GPU 的 SIMD 执行模型中，同一 warp 内分支发散的代价是什么？什么样的 `if` 实际无害？

**解答：**

**执行模型**：GPU 把相邻的 32/64 个线程（warp / wavefront）锁在同一个指令指针上——同一拍所有线程执行**同一条指令**，各自操作各自的数据（SIMT）。

**发散的代价**：当 warp 内有的线程走 if、有的走 else，硬件无法分头执行，只能**串行走完两条路径**：先执行 if 分支（else 侧线程被掩码挂起、白白陪跑），再执行 else 分支（if 侧挂起）。耗时 = 两分支之和，发散期间有效算力减半；嵌套发散按层数继续恶化。空间上，发散与否取决于**屏幕相邻像素/相邻顶点是否做出相同选择**——条件在空间上越破碎，发散越频繁。

**实际无害的三类 if：**

1. **被编译为选择指令的**：`x = cond ? a : b`、`mix(a, b, step(...))`，以及两侧都是廉价表达式的短 if——编译器生成无跳转的 conditional select，两侧都算然后选一个，根本没有“分支”。这也是指南推荐 mix/step 写法的真正原因：不是 if 关键字有毒，而是**控制流跳转**有成本，而 mix/step 把控制流降级成了数据流。
2. **warp 内一致的（uniform branch）**：条件只依赖 uniform（如 `if (u_enableFog)`）——全部线程必然同走一边，硬件直接整体跳转，零发散代价。用 uniform 开关调试特性是免费的。
3. **空间上大块连贯的**：条件虽依赖 varying 但在屏幕上成片相同（如“距离 > 雾起点”——以雾界线为边界，两侧各自大片一致），只有边界处的少数 warp 发散，整体代价可忽略。

真正要避免的是**高频空间破碎的分支**：每个像素按哈希/噪声独立选路（如 `if (hash(worldPos) > 0.5) {复杂路径A} else {复杂路径B}`），几乎每个 warp 都全额支付双倍代价。判断口诀：先问“相邻线程会不会选不同的路”，再问“两条路各有多贵”。

### Q3-13 雾效解决什么问题；fogEnd 与清屏色的约束

> 雾效在本项目里解决的核心体验问题是什么？fogEnd 与 R·CHUNK_SIZE 应满足什么关系？背景清屏色取什么值？

**解答：**

**核心问题：遮蔽世界的物理边缘。** 活跃窗口是有限的——站在窗口中心，世界在 ~R·CHUNK_SIZE 距离处戛然而止。没有雾时，玩家直视远方就能看到“世界尽头的悬崖”，且移动时新区块在视野尽头**凭空蹦出**、旧区块凭空消失（pop-in/pop-out）——“无限世界”的幻觉当场穿帮。雾效让可见度在到达数据边缘**之前**平滑衰减到零：区块的生灭全部发生在浓雾深处，玩家的感知是“世界一直延伸进雾里”。这是用一个图形技巧补全架构语义的典型案例——滑动窗口负责“数据上无限”，雾负责“**观感上无限**”，缺一不可。（附带收益：远处片元混入大量雾色后，LOD/动画的远端瑕疵也被掩盖。）

**fogEnd 的约束**：必须 ≤ 窗口的**最小保证可视距离**。切比雪夫窗口对摄像机的保证距离由“面方向”决定：摄像机在中心区块内任意位置时，到窗口边界面的最小距离 = `R × CHUNK_SIZE`（R=3、CHUNK_SIZE=4 时 = 12；角方向虽远达 √3 倍但不能指望）。再留出余量：

```
fogEnd ≤ R × CHUNK_SIZE − CHUNK_SIZE（一个区块的安全边距）   → 本例 ≈ 10~11
fogStart ≈ 0.5 × fogEnd（过渡带足够宽才显得自然）
```

若 fogEnd 超过保证距离，浓雾尚未完全遮蔽处就有区块生灭，pop-in 重新可见。反过来，这条不等式也是**调参的主从关系**：想看得更远，先加大 R（付出 Q3-6 的三次方代价），fogEnd 跟随；而不是单独拉 fogEnd。

**清屏色必须 = 雾色**（且雾色用同一 uniform 喂给 clearColor 与着色器）。推导：视线方向没有任何几何体的像素显示的是清屏色，而有几何体但极远的像素显示的是 ~100% 雾色——二者在视觉上同属“无穷远”，颜色必须一致，否则天际线处出现几何体轮廓的“剪影”，雾的伪装当场被拆穿。常见错误是清屏用黑色、雾用灰蓝——满屏立方体浮在黑底上的诡异画面。

## 阶段四：漫游摄像机系统与 CPU 端裁剪交互

### Q4-1 pitch 不钳位的完整故障链路

> 不钳位时第一帧异常出现在哪个具体函数的哪一步？从「屏幕黑了」反向推出完整链路。

**解答：**

**正向故障链**（pitch 被鼠标推到 ±π/2 的那一帧）：

1. `dir = (cos(pitch)sin(yaw), sin(pitch), cos(pitch)cos(yaw))`：pitch = π/2 时 cos(pitch) = 0 → dir = **(0, 1, 0)**，与 worldUp 完全平行。方向向量本身仍合法——此步不出错，错在下一步；
2. `lookAt` 第二步：`r = cross(f, worldUp) = cross((0,1,0), (0,1,0)) =` **(0, 0, 0)**；
3. `normalize((0,0,0))`：计算 `1/√0` → **除以零 → 分量全 NaN**；
4. `u = cross(r, f)`：NaN 输入 → NaN 输出；视图矩阵 16 个元素污染殆尽（NaN 参与任何运算仍是 NaN——它有极强的传染性）;
5. 顶点着色器：`P · V · M · position` → 所有顶点裁剪坐标 = NaN；
6. 光栅化器对含 NaN 的图元的处理是**直接丢弃**（NaN 与任何比较都为 false，裁剪测试不可能通过）→ 零片元产生；
7. 屏幕呈现：只剩 `gl.clear` 的底色——**整屏内容消失，且控制台零报错**（NaN 是合法浮点值，不触发 GL 错误）。

**反向排查路径**（这才是该问题的训练价值——黑屏时如何系统逼近）：屏幕黑但 clear 色还在 → 说明管线在跑、几何全军覆没 → 按附录 A 第 3 条打印视图矩阵 → 发现 NaN → NaN 只能来自 0/0、√负数、0×∞ → 视图矩阵的构造链上唯一的除法在 normalize → 哪个 normalize 输入了零向量？→ cross(f, up) 模长为零 ⇔ f ∥ up ⇔ pitch = ±90° → 检查输入日志确认俯仰到顶 → 根因落定。**修复在源头**：钳位 `pitch ∈ [−π/2+ε, π/2−ε]`（ε ≈ 0.02 rad，既防 NaN 也防“接近平行”的数值抖动，见 Q1-11）；normalize 内部的零向量保护只作为第二道防线。

### Q4-2 指数趋近为什么比线性插值稳健

> 为什么 `v += (target − v) * (1 − exp(−k·dt))` 比 `v += (target − v) * k * dt` 更稳健？后者何时发散？

**解答：**

两者都想实现同一个物理意图：速度向目标速度按比例衰减逼近，即微分方程 `dv/dt = k(target − v)`，精确解为指数趋近 `v(t) = target + (v₀ − target)·e^(−kt)`。

**线性形式是该方程的显式欧拉积分**，把连续衰减近似成离散大步。设误差 `e = target − v`，每步 `e ← e·(1 − k·dt)`，稳定性完全取决于步长：

- `k·dt < 1`：正常收敛；
- `k·dt = 1`：一步精确到达（侥幸）；
- `1 < k·dt < 2`：**过冲振荡**——每步冲过头再折返，速度围绕目标值来回摆，手感“橡皮筋抖动”；
- `k·dt > 2`：|1 − k·dt| > 1，误差**每步放大，指数发散**——速度数值爆炸。

代入实际数字：k = 10（指南推荐值）在 60 FPS（dt ≈ 0.017）时 k·dt = 0.17，安全；但**一次卡顿** dt = 0.25 秒 → k·dt = 2.5 → 当帧直接进入发散区。也就是说线性形式埋着一颗“掉帧即引爆”的雷，而掉帧恰恰是最不该引发次生灾害的时刻。

**指数形式是精确解的直接采样**：衰减因子 `e^(−k·dt) ∈ (0, 1]` 对**任意** dt 恒成立——dt 再大，最坏情况也只是“一步走到 target”，**无条件稳定，物理上不可能过冲**。同时它保证严格的帧率无关性：60 FPS 走两帧与 30 FPS 走一帧，结果按定义完全一致（`e^(−k·dt₁)·e^(−k·dt₂) = e^(−k(dt₁+dt₂))`，指数的可加性）；线性形式只在 dt→0 时近似满足此性质，不同帧率下手感有可测差异。结论：每帧一次 `exp` 的成本可忽略，换来无条件稳定 + 严格帧率无关，没有理由用欧拉近似。（钳位 deltaTime ≤ 100 ms 仍要做——它防的是位置积分的瞬移，与本题的速度稳定性互补。）

### Q4-3 失焦不清空按键状态的后果与根因

> 失焦时不清空按键状态会发生什么？从浏览器事件模型层面解释根因。

**解答：**

**现象**：按住 W 前进时 Alt+Tab 切走，在别的窗口里松开 W，切回来——人物**永久自动前进**，按任何键都停不下来（直到再按一次 W 触发新的 keydown/keyup 周期）。测试时偶现、难复现，用户报障描述为“角色有时候自己乱跑”。

**事件模型层面的根因**：键盘事件只派发给**当前持有焦点的文档**。状态字典方案的本质是用 keydown/keyup **事件对**重建“按键此刻是否按下”这一物理状态——它隐含一个假设：**每个 keydown 必有配对的 keyup 到达**。失焦恰好击穿该假设：keydown 在失焦前到达（字典置 true），keyup 发生在失焦后——浏览器把它派发给新焦点窗口（或丢弃），你的页面**永远收不到**。字典里的 `W: true` 成为无人负责的孤儿状态，软件状态与物理键盘状态从此脱钩。同类丢失场景还有：指针锁定退出瞬间、系统级快捷键拦截（Alt+Tab 的 Alt 本身就常丢 keyup）、浏览器弹窗抢焦点。

**修复**：监听 `window.blur`、`document.visibilitychange`（隐藏时）、以及 `pointerlockchange`（锁定丢失时），三处统一调用 `keyState.clear()`。代价是“切回来需重新按键”——正确且符合直觉。**通用教训**：凡是靠“成对事件”维护的镜像状态（键盘、鼠标按下、触摸点），都必须为“配对事件永不到达”准备重置路径；边界事件（失焦、页面隐藏）就是天然的重置时机。

### Q4-4 事件回调直改摄像机 vs 主循环统一处理

> mousemove 一帧可能触发多次，「回调中直接改摄像机」与「主循环统一处理」行为有何差异？后者好在哪里？

**解答：**

背景事实：高轮询率鼠标（如 1000 Hz）在一个 16.6 ms 帧内可产生十几个 mousemove 事件。两种方案对比：

**回调直改**：每个事件到达时立即 `yaw += dx·s; pitch = clamp(...)`。摄像机状态在一帧**中途多次变化**，由此产生三类问题：

1. **帧内状态不一致**（最实质的危害）：一帧的渲染流程会多处读取摄像机——剔除用它算视锥、世界管理用它算所在区块、最后构造视图矩阵。若 mousemove 恰好插在剔除与构造视图矩阵之间，**剔除用的是旧朝向、渲染用的是新朝向**——视锥边缘的区块被错误剔除，表现为转头瞬间屏幕边缘“闪缺一块”。偶发、与输入时序相关、无法稳定复现——最恶劣的 bug 画像。
2. **逻辑分散**：灵敏度、钳位、平滑等处理散落在事件回调里，与 deltaTime 相关的逻辑（如朝向阻尼）根本无法在回调中正确实现（回调没有稳定的 dt 概念）。
3. **不可测试/不可回放**：行为依赖事件到达的真实时序，无法用录制的输入序列做确定性回放。

**主循环统一处理**：回调只做最廉价的累积——`accumDx += e.movementX`；每帧开头一次性取走累积量、清零、应用到 yaw/pitch 并钳位。收益逐条对应：① 摄像机状态在“输入应用”之后、到帧结束**冻结不变**，剔除与渲染读到同一姿态，帧内一致性由结构保证；② 所有输入处理集中一处，灵敏度/平滑/钳位按帧逻辑实现；③ 输入退化为“每帧一个 (dx, dy) 数据”，可录制、可重放、可单元测试。

注意这与 Q4-3 同属一个架构原则：**事件回调是边界层，只负责采集与缓存；一切状态演化集中在主循环的固定阶段**。这是游戏引擎“输入采样 → 仿真 → 渲染”分阶段管线的最小实践。多次累积后统一钳位与逐次钳位在数学上等价（clamp 的单调性），不必担心行为差异。

### Q4-5 距离剔除应在生成时还是每帧写入时

> 距离剔除应该发生在区块生成时还是每帧写入时？为什么？这与槽位/上传缓冲分离设计的关系？

**解答：**

**必须在每帧写入（压缩拷贝）时。** 判据是数据时效性：剔除条件 `distSq(实例, 摄像机) < 4` 的两个输入中，实例位置是静态的，**摄像机位置每帧都变**——条件的值每帧都可能翻转。若在区块生成时剔除（生成的一瞬间摄像机恰好在旁边，就把附近实例永久排除出槽位数据），会产生两类持久错误：摄像机**离开**后，那些被剔除的立方体永远缺失（地上一个永久的洞，直到区块重新生成）；摄像机**走近**生成时不在身边的区块，半径 2 内的立方体又不会被剔除（硬需求失效）。错误会以“摄像机生成时刻的历史位置”的形状**烙印**在世界里——又一个时序相关、难复现的 bug。

**与两层缓冲设计的对应关系**（这正是 Q3-2 架构的另一半理由）：

- **槽位/影子缓冲 = 时不变数据**：只由区块坐标决定（确定性生成），生成一次、跨帧复用，绝不掺入任何与摄像机相关的信息;
- **上传缓冲 = 时变视图**：每帧由影子数据经“当帧摄像机”过滤（距离剔除 + 视锥剔除）压缩而成，用完即弃。

剔除属于“过滤”，天然归属第二层。一般化的设计准则：**给每份数据标注它依赖的输入的变化频率，依赖不同频率输入的计算必须放在不同的更新通道里**——把每帧变化的条件混进低频数据，得到的就是上述“历史烙印”类 bug；反过来把低频计算放进每帧通道，则是白白烧 CPU（让生成逻辑每帧重跑）。

### Q4-6 discard 的决策框架：为什么植被又普遍用它

> 既然 discard 有害，为什么透明植被渲染又普遍使用 discard？提炼「在管线哪一级剔除」的决策框架。

**解答：**

**两个场景的本质差异在“可见性信息在哪一级才可获得”：**

- 本项目剔除半径 2 内的立方体：可见性以**实例为单位**整体成立或不成立，且仅由两个 CPU 已知量（实例位置、摄像机位置）决定——CPU 在提交前就有完整信息，当然在最早、最便宜的位置（不写入缓冲，0 顶点 0 片元成本）解决。
- 植被（草、树叶用一张带 alpha 的四边形贴图）：一个面片上**哪些像素属于叶子、哪些是透明背景**，由纹理采样结果决定——这个信息**只在片元着色器采样纹理之后才存在**。CPU 无法替代（除非把每片叶子建成真实几何，成本反而爆炸）。逐像素的剔除只有两条路：alpha blending（需要按深度排序、不写深度，对成千上万株植被是排序灾难）或 **alpha test（discard）**——付出失去 Early-Z 的代价，换取免排序、正确写深度。两害相权，discard 是植被场景的正解。

**决策框架——“在拥有足够信息的最早一级剔除”：**

| 管线层级 | 剔除手段 | 单位成本 | 适用前提 |
|---|---|---|---|
| CPU 提交前 | 距离/视锥/遮挡剔除 | 整个物体归零 | 可见性由 CPU 已知量按物体粒度决定 |
| 顶点/图元级 | 背面剔除、视锥裁剪（硬件自动） | 免费 | 硬件内建，保证开启即可 |
| Early-Z（片元着色前） | 深度预剔除 | 跳过片元着色 | 不用 discard、不改 gl_FragDepth |
| 片元着色中 | discard | 已付全部片元成本 | 可见性必须等纹理/逐像素计算才知道 |

两条推论：① **能上移一级就上移**——每提前一级，被剔除单元的成本就少付一段管线；② discard 不是“禁用”而是“最后手段”，使用时应把代价局部化：把需要 discard 的物体放进**单独的 draw call/着色器**，让主体几何（本项目的百万立方体）保持无 discard、Early-Z 全速的路径，别让少数植被拖累整个场景的深度优化。

### Q4-7 视锥剔除的粒度选择与 p-vertex 技巧

> 视锥剔除应该逐实例还是逐区块？AABB 与平面的快速测试（p-vertex 技巧）如何做？

**解答：**

**粒度选择——逐区块（两级剔除）。** 成本账：逐实例 = 21,952 实例 × 6 平面 ≈ **13 万次**点-平面点乘/帧；逐区块 = 343 区块 × 6 平面 ≈ **2,000 次** AABB-平面测试，砍掉 98% 的测试量，而损失的精度仅是“区块部分在锥内时，其锥外实例被陪绑绘制”——单区块才 64 个实例，GPU 的硬件裁剪会无声处理它们，陪绑成本远低于 CPU 逐实例测试的成本。流程编排：每帧先用视锥对 343 个区块粗筛（区块 AABB 已知：基址 + CHUNK_SIZE 立方），通过的区块才进入逐实例距离剔除（Q4-5）+ 压缩写入。典型视角（fov 70°）下视锥只覆盖窗口的 1/4~1/3 区块，**剔除后的实例数、顶点量同步降到 1/3**——这是本项目单笔收益最大的 CPU 优化。

**p-vertex（最正顶点）技巧**——AABB 对单平面 `n·p + d ≥ 0`（法线指向锥内）的测试只需测**一个点**：

```
AABB 由 (min, max) 定义，对平面法线 n：
p-vertex（最正顶点）= 在 n 的每个分量上取更“顺法线”的那端：
  px = (n.x ≥ 0) ? max.x : min.x     // y、z 同理
若 n·p_vertex + d < 0：连盒子里最靠锥内的点都在平面外 → 整盒在外，剔除；
否则该平面测试通过。
```

原理：AABB 上 `n·p` 的最大值必在这个角点取得（逐分量独立最大化），它在平面内 ⇔ 盒子至少有一点在平面内。每平面成本 = 3 次符号判断 + 1 次点乘。六个平面有一个判“全外”即剔除该区块；全部通过则保留（含少量假阳性——盒子在所有平面内侧却在锥角之外的情况，结果只是多画无害）。

**两个必踩的坑**：① 从 `P·V` 提取的平面系数**未归一化**就用——对“在内/在外”的符号判断其实无影响（齐次缩放不变号），但一旦你想用距离值（如按距离做 LOD）就错了，养成提取后归一化的习惯；② 列主序数组里“取第 i 行”的下标是 `m[i], m[i+4], m[i+8], m[i+12]`——行列又一次容易搞反，写完立刻用“冻结视锥”调试模式（指南步骤 4）做可视化验证。

## 阶段五：工程化重构与数学文档规范

### Q5-1 哪里值得抽象成类；统一判断标准

> 本项目中哪里反而值得抽象成类？判断「值得/不值得抽象」的统一标准是什么？

**解答：**

**值得成类的，与各自要保护的不变式：**

- **`ShaderProgram`**（最典型）：它的不变式是“暴露给外界的实例必然处于编译成功、链接成功、location 缓存与当前 program 一致的状态”。构造函数把“编译→检错→链接→检错→枚举 uniform”锁成不可拆分的序列，外界拿到实例即拿到保证；若用裸函数+裸 WebGLProgram，每个调用点都可能拿到半成品。状态缓存（当前 use 的是谁）也需要一个归属地。
- **`SlotAllocator`**：不变式是“每个槽位要么在 free list、要么被恰好一个区块持有，无重复无遗漏”。封装成类后，外界只能通过 `alloc()/free()` 改变状态，双重释放、凭空分配在接口层就被堵死；裸数组 + 散落各处的 push/pop 则任何一处手滑都破坏全局。
- **`Camera` / `WorldManager`**：有跨帧存续的内聚状态（位置朝向 / 活跃集合）与围绕状态的操作，成类（或闭包模块）自然。

**不值得的**：`vec3/mat4`——数据是裸 `Float32Array`（GL 接口的要求），函数无状态，没有任何不变式需要守护，包成 `Vector3` 类只会引入每次运算的对象分配（违背零分配）与 this 调度开销；`CubeGeometry` 介于两者之间——它有“VAO 配置完整”这一构造期不变式，轻量类合理，但不必再为它设计继承体系。

**统一标准：是否存在需要保护的不变式（invariant）。** 类的本质价值是把“状态 + 维持状态合法性的规则”关进一个边界，使外界**无法构造或观察到非法状态**。有不变式 → 封装是刚需；没有 → 类只是仪式，模块级纯函数更简单、更可测、更快。反向信号也成立：一个类如果所有字段都是 public、任何方法序列都合法，它就没有不变式，应当退化为函数；一组散落的函数如果都在小心翼翼维护同一份数据的某种一致性，它们就该被收进一个类。

### Q5-2 R 从 3 升到 5 的增长量；瓶颈三分法实验

> R 从 3 升到 5，实例数增长多少倍？如何设计实验区分瓶颈在顶点、片元还是 CPU 写入？

**解答：**

**增长量**：区块数从 7³ = 343 增至 11³ = 1,331（**3.88 倍**），实例上限从 2.2 万增至 **8.5 万**，顶点量 ~204 万/帧。注意帧时间**不会**等比增长 3.88 倍——各管线阶段的负载弹性不同（片元数主要由屏幕分辨率与 overdraw 决定，受 R 影响远小于顶点数），所以“调 R 前后各测一次”本身就是一次免费的瓶颈定位。

**三分法实验**（控制变量，每次只动一个；以下“变/不变”均指 p95 帧时间）：

1. **测片元**：渲染分辨率减半（canvas 物理尺寸 ×0.5，CSS 尺寸不动），片元数降到 1/4，顶点与 CPU 完全不变。帧时间显著下降 → **片元瓶颈**（填充率/overdraw），对策：减 overdraw、简化片元着色器、降分辨率；几乎不变 → 排除片元，继续。
2. **测顶点**：分辨率复原，实例数减半（限制 instanceCount 即可，CPU 照常写入全量——刻意只动 GPU 顶点量）。帧时间近半 → **顶点瓶颈**，对策：视锥剔除做实（Q4-7）、面消隐（附录 C-6）、LOD;不变 → 继续。
3. **测 CPU**：Performance 面板看主线程——若每帧 JS（剔除循环 + 写入 + bufferSubData）耗时逼近帧预算，或 GPU 侧出现空等，则 **CPU 瓶颈**，对策：剔除循环去分配/去函数调用开销、两级剔除减少候选集、生成移 Worker。辅助判据：CPU 瓶颈时降低分辨率帧率纹丝不动（GPU 在等 CPU 投喂）。

**两条纪律**：① **一次只动一个变量**——同时改分辨率和实例数，结果无法归因；② 每个实验记录进性能档案（机器、参数、p95 数值），它们就是阶段五要求的基准线。另外注意瓶颈会**迁移**：解决片元瓶颈后顶点可能成为新瓶颈，优化是“测量→定位→消除→再测量”的循环，而非一次性诊断。
