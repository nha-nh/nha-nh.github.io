# WebGL 2.0 无限三维世界 —— 底层图形学与前端工程化实战指南

本技术文档引导你从零构建一个基于 WebGL 2.0 的无依赖、高性能、**真实无限**的三维世界。世界在空间上无硬边界：随着摄像机移动，前端持续生成新的网格区块，后方不可见的区块即时销毁，使得世界在观感上永远延伸，而内存与绘制负载保持恒定。

文档**不提供任何具体实现代码**，而是作为工程路线图与思维导图，通过架构约束、核心逻辑拆解以及引导性问题，帮助你独立推导、设计并完成项目。

### 核心术语表

| 术语 | 含义 |
|---|---|
| Chunk（区块） | 边长 `CHUNK_SIZE` 的立方体空间单元，是生成/销毁的最小粒度 |
| 活跃半径 R | 以区块为单位，摄像机周围保持加载的切比雪夫距离 |
| 实例（Instance） | 一个被绘制的立方体，仅以一个 4×4 变换矩阵的形式存在于 VBO 中 |
| 槽位（Slot） | 实例 VBO 中预划分的、供单个区块写入的连续空间段 |
| NDC | 归一化设备坐标，WebGL 中三轴范围均为 [-1, 1] |
| 列主序 | 矩阵在内存中按列连续存储，WebGL/GLSL 的原生约定 |

## 阶段零：全局架构与“无限世界”形式化定义

在写第一行代码前，必须严格定义什么是“无限世界”，以及整体数据流如何支撑这一概念。本阶段产出物是**一页纸的架构图**和**一组常量的初始取值**。

### 0.1 世界的边界定义

* **无限 ≠ 无穷大数据**
  世界在任何时刻只存在有限个“活跃区块”（Chunk）。设摄像机的观测半径为 `R` 个区块，则活跃区块总数 ≈ `(2R+1)³`。该数量与玩家已探索的空间大小无关——哪怕移动到坐标 (10⁶, 0, 10⁶)，活跃区块数依然不变。
* **确定性生成**
  同一个区块坐标 `(cx, cy, cz)` 无论何时被加载，其内容必须完全一致（立方体的位置、颜色、动画相位都相同）。这意味着区块内容必须是**坐标的纯函数**——由坐标经哈希/噪声函数推导，而非随机生成后存储。这是“销毁后可凭空重建”的前提，也是整个内存恒定方案的理论根基。
* **动态生成与销毁**
  每帧（或每隔 N 帧）根据摄像机当前所在区块坐标，计算哪些区块新进入“活跃半径”，哪些已移出。对新出现的区块，分配或复用实例缓冲区槽位；对离开的区块，将其槽位标记为空闲，供后续复用。**不存在全局坐标遍历，没有无限增长的数据结构。**
* **无状态实体**
  每个立方体不在 CPU 端保存独立对象（无类实例）。所有运动、颜色、缩放动画均由着色器根据**世界坐标 + 统一时间变量**实时计算。

*引导问题：*
1. *【Q0-1】若 R = 3、CHUNK_SIZE = 4，活跃区块数与最大实例数各是多少？每个实例一个 mat4（64 字节），实例 VBO 的总容量是多少 MB？这个数字告诉你什么？*
2. *【Q0-2】“确定性生成”意味着什么函数签名？尝试写出伪签名：`generateChunk(cx, cy, cz) → 实例矩阵列表`。它能接收任何随时间变化的参数吗？*
3. *【Q0-3】为什么“无状态实体”和“确定性生成”互为充要？如果立方体允许被玩家修改（如挖掉一个），架构需要增加什么？（提示：差异表 / 稀疏覆盖层——本项目不做，但要知道边界在哪。）*

### 0.2 浮点精度与“伪无限”的边界（容易被忽视）

* 世界坐标使用 32 位浮点。当摄像机移动到 10⁵ 量级时，`Float32` 的尾数精度只剩约 1 个单位级别，顶点会开始“抖动”（vertex jitter）。
* *引导问题【Q0-4】：`Float32` 在数值 100,000 附近的最小可分辨增量约是多少？（提示：2⁻²³ × 2¹⁷ ≈ 0.0156）这对相机平滑移动意味着什么？*
* *进阶方向（暂不实现，但架构上要预留认知）：「浮动原点」（Floating Origin）——周期性把整个世界平移回原点附近，摄像机坐标用双精度或「区块整数坐标 + 区块内浮点偏移」两段式表示。在着色器中用 `相对摄像机坐标` 而非绝对世界坐标计算，可彻底回避该问题。*

### 0.3 渲染数据流（每帧）

```
输入采集 ──→ 摄像机更新 ──→ 世界管理 ──→ CPU 剔除 ──→ 缓冲区上传 ──→ Draw Call ──→ (下一帧)
 键鼠状态     位置/朝向      区块进出      距离+视锥      bufferSubData   instanced
```

1. **输入**：摄像机位置、欧拉角、系统时间、键鼠状态。
2. **世界管理**：计算摄像机所在区块，更新活跃区块集合，生成/回收实例变换数据。
3. **CPU 剔除**：剔除摄像机周围半径 2 以内的个体（硬性需求），以及视锥体外的个体/区块。
4. **缓冲区上传**：将筛选后的实例变换矩阵写入动态 VBO。
5. **单次 Draw Call**：实例化渲染一次绘制所有剩余立方体，着色器完成顶点动画、光照。
6. **呈现**：浏览器合成。

*引导问题：*
1. *【Q0-5】上述每一步的耗时上限各应是多少，才能保证 60 FPS（16.6 ms 预算）？给每一步分配一个预算（如世界管理 ≤ 2 ms）。*
2. *【Q0-6】哪些步骤是每帧必须执行的？哪些可以摊销（如世界管理可在摄像机跨区块时才触发）？*
3. *【Q0-7】数据在哪些环节穿越了 CPU/GPU 边界？为什么要把穿越次数压到最少？*

### 0.4 模块划分与目录结构建议

```
src/
  math/        vec3.ts  mat4.ts          ← 纯函数，零依赖，可单测
  core/        gl-context.ts  shader-program.ts  geometry.ts
  world/       world-manager.ts  chunk-generator.ts  slot-allocator.ts
  camera/      camera.ts  input.ts
  render/      renderer.ts  frustum.ts
  main.ts      ← 主循环，唯一知道所有模块的地方
```

* **依赖方向必须单向**：`math` 不依赖任何模块；`world` 不依赖 `render`（它只产出 `Float32Array`，不知道 GL 的存在）；只有 `main.ts` 和 `renderer` 接触 GL 上下文。
* *引导问题【Q0-8】：为什么 `WorldManager` 不应该持有 `gl`？当你想为区块生成编写单元测试、或将来把它移入 Web Worker 时，这个约束分别带来什么好处？*

## 阶段一：纯函数高性能数学库（Math Core）

### 目标
所有向量/矩阵运算均采用无状态纯函数，数据载体使用 `Float32Array`，通过 `out` 参数外置内存管理，在整个引擎稳态运行期间**零堆分配**。

### 为什么是 `out` 参数而不是返回新对象？

*逻辑拆解：*
1. JS 引擎的分代 GC 对“每帧上万个短命小对象”的场景极不友好——即使 minor GC 很快，也会造成帧时间尖刺（卡顿一帧远比平均慢 10% 更可感知）。
2. `out` 模式把分配决策权交还调用者：调用者可在初始化时一次性预分配，运行期只做写入。
3. 代价是 API 易用性下降、出现别名（aliasing）风险。这是性能与可读性的有意识权衡，应写入文档。

*引导问题【Q1-1】：打开浏览器 Performance 面板，分别录制「每帧 new 一万个 Float32Array」与「复用预分配数组」两个最小实验，对比 GC 轨迹。预期分别看到什么样的内存曲线？为什么帧时间尖刺比平均帧时间更值得关注？*

### 步骤 1：三维向量工具集（`vec3.ts`）

* **运算规范**
  - `add(a, b, out)`、`subtract(a, b, out)`、`scale(a, s, out)`：结果写入外部提供的 `out`，调用者负责预分配。
  - `dot(a, b)` 返回标量，不涉及分配。
  - `cross(a, b, out)` 同样使用 `out`。
  - `normalize(a, out)`：注意零向量保护，避免除零产生 NaN 传播导致整个场景闪烁或消失。
  - `lengthSq(a)`：平方长度，供距离比较使用（见阶段四）。
* **别名安全**
  *引导问题【Q1-2】：`cross(a, b, a)`（out 与输入重叠）会发生什么？如何在不引入堆分配的前提下修复？*
  *引导：叉乘的每个输出分量都依赖输入的多个分量，思考写入顺序与读取顺序的冲突。为每个有此风险的函数写一个别名测试用例。*
* **暂存器策略**
  *引导问题【Q1-3】：每帧可能调用上万次向量运算，中间结果放哪里？模块级暂存向量（如 `tmp0`, `tmp1`）的使用契约应包含哪些禁令？*
  *引导：思考暂存器与异步回调、递归、Web Worker 的关系；思考「调用方自带暂存」与「模块级共享暂存」各自的利弊。*
* **几何语义思考**
  *引导问题【Q1-4】：`dot(forward, toTarget) > 0.999` 在几何上意味着什么角度范围？点乘的符号、叉乘结果的模长各有什么几何语义？它们分别能在本项目哪些环节派上用场？*

### 步骤 2：列主序 4×4 矩阵工具集（`mat4.ts`）

* **存储约定**
  矩阵元素 `m[0..15]` 按列主序存储：列 0 (m0-m3)，列 1 (m4-m7)，列 2 (m8-m11)，列 3 (m12-m15)。
  数学记号 `M[row][col]` 对应 `m[col*4 + row]`。**在文件头注释中画出这张映射表**——后续 90% 的矩阵 bug 都源于行列混淆。
  *自检问题【Q1-5】：平移分量 (tx, ty, tz) 在数组中是哪三个下标？答错说明什么？*
* **核心函数清单**
  - `identity(out)`
  - `translate(m, tx, ty, tz, out)` / `rotateX/Y/Z(m, angle, out)` / `scale(m, sx, sy, sz, out)`
  - `multiply(a, b, out)` — 必须处理 `out === a || out === b` 的别名情况：用 16 个局部标量暂存结果再写回。
  - `perspective(fovY, aspect, near, far, out)`
  - `lookAt(eye, center, up, out)`
  - （可选）`invert(m, out)`、`transpose(m, out)`
* **复合顺序推导（亲手推导）**
  *引导问题【Q1-6】：要实现物体先绕自身 Y 轴旋转 β，再绕世界 X 轴旋转 α，最后平移到 (Tx, Ty, Tz)，矩阵连乘顺序是怎样的？如何用一个具体点验证？*
  *引导：设物体局部坐标点 p，思考「从右往左读：最先作用的变换离 p 最近」。每个旋转函数都至少写一个“90° 转特殊点”的单元测试。*
* **法线与逆转置矩阵**
  *补充：法线是协变向量，变换需要世界矩阵的逆转置 `(M⁻¹)ᵀ`。若变换仅含旋转与平移，M 的 3×3 部分正交，逆转置等于自身，可直接使用；含均匀缩放时法线只是长度变化，归一化即可修正；只有**非均匀缩放**才真正需要逆转置。*
  *引导问题【Q1-7】：本项目的缩放动画发生在着色器内（阶段三），CPU 端矩阵只含旋转 + 平移。那么法线矩阵需要逆转置吗？着色器内的顶点位移动画会破坏法线正确性吗？哪些动画破坏、哪些不破坏？*
* **单元测试建议**
  数学库是全项目唯一能脱离 GPU 验证的部分，务必覆盖：单位阵性质（M·I = M）、乘法结合律抽样验证、`M · M⁻¹ ≈ I`、别名调用、`lookAt` 输出乘以 eye 点应得原点。

### 步骤 3：核心空间投影与视图变换推导

* **透视投影矩阵推导（亲手推导，无需查表）**
  给定 fovY、aspect、near、far：
  1. Y 方向缩放：`f = 1 / tan(fovY / 2)` —— 由「相似三角形：视平面高度与 z 的比值恒定」推出。
  2. X 方向缩放：`f / aspect`。
  3. Z 方向：将 `[-near, -far]`（视空间，看向 -Z）非线性映射到 `[-1, 1]`（NDC）。常用形式：
     `m[10] = -(far+near)/(far-near)`，`m[14] = -2·far·near/(far-near)`，`m[11] = -1`。
  4. **验证**：代入 `z = -near` 应得 NDC -1，`z = -far` 应得 NDC +1。把这一验证写进单元测试。
  *引导问题【Q1-8】：`m[11] = -1` 的作用是什么？如果 near 设为 0 会发生什么？*
* **深度精度分析**
  *引导问题【Q1-9】：为何远处的 Z-fighting 更严重？反向 Z（Reversed-Z）为什么能缓解？在 WebGL 2.0 中实现反向 Z 需要改动哪几处？*
  *实践引导：先用 near=0.01、far=10000 制造一次 Z-fighting，亲眼看到问题，再把 near 提到 0.1 观察改善——「先复现问题再优化」是图形调参的基本素养。*
* **LookAt 正交基推导（亲手推导）**
  ```
  f = normalize(center - eye)        // Forward（指向目标）
  r = normalize(cross(f, worldUp))   // Right（注意叉乘顺序与你的手系约定一致）
  u = cross(r, f)                    // Up（重新正交化）
  ```
  视图矩阵的本质：**把世界坐标变换到“以摄像机为原点、以 r/u/-f 为轴”的坐标系**，即摄像机世界变换的逆。由于旋转部分正交，逆 = 转置；平移部分则是 `-dot(基向量, eye)`。
  *引导问题【Q1-10】（亲手推导）：为何视图矩阵的平移部分是负点乘 `-dot(r, eye)` 而不是简单的 `-eye`？*
  *引导问题【Q1-11】：当 f 与 worldUp 平行（垂直俯视）时，上述构造过程在哪一步退化？产生什么后果？这与阶段四的哪个设计决策相关？*

## 阶段二：WebGL 2.0 基础管线与显式封装（Render Core）

本阶段终点是一个**可旋转观察的、带 Phong 光照的单个立方体**。不要在没有这个“最小可见成果”之前就去写实例化和区块系统——那等于在黑暗中叠加三层不确定性。

### 步骤 0：上下文、画布与主循环（常被跳过、必须显式设计）

* **获取上下文**
  - `canvas.getContext('webgl2', options)`，检查返回 null 的情形并向用户呈现降级提示。
  - *引导问题【Q2-1】：上下文选项 `antialias`、`alpha`、`powerPreference` 各应取什么值？`alpha: false` 为什么通常能提升合成性能？*
* **画布尺寸与 DPR**
  - CSS 尺寸（布局像素）与绘制缓冲尺寸（物理像素）是两件事。监听尺寸变化（推荐 `ResizeObserver`），把 `canvas.width/height` 设为 `clientSize × devicePixelRatio`，并同步调用 `gl.viewport` 与更新投影矩阵的 aspect。
  - *常见陷阱：忘记乘 DPR → 高分屏上画面模糊；在每帧无条件设置 `canvas.width` → 隐式清空画布且强制重建缓冲，性能灾难。*
* **全局状态初始化**
  - `gl.enable(gl.DEPTH_TEST)`；`gl.enable(gl.CULL_FACE)` 并确认立方体三角形绕序为 CCW。
  - *引导问题【Q2-2】：背面剔除能省掉立方体约几成的片元工作？如果开启后立方体「破了几个洞」，说明什么？*
* **主循环与时间步**
  - `requestAnimationFrame` 驱动；用相邻两帧时间戳之差得到 `deltaTime`，并对其钳位（如 ≤ 100 ms），防止切后台回来后物理瞬移。
  - 动画时间 `u_time` 应累积自钳位后的 deltaTime，而非直接用绝对时间戳。
  - *引导问题【Q2-3】：为什么 `u_time` 不能直接用绝对时间戳？长时间运行后动画为什么会“变糙”？对策是什么？*
* **上下文丢失（工程健壮性）**
  - 监听 `webglcontextlost`（preventDefault + 停止渲染循环）与 `webglcontextrestored`（重建所有 GPU 资源）。
  - *引导问题【Q2-4】：上下文恢复后哪些对象需要重建？哪些不需要？这与阶段零的哪条架构约束相互印证？*

### 步骤 1：`ShaderProgram` 类设计

* **职责**：封装单个着色器程序的生命周期，提供类型安全的 Uniform 更新接口。
* **封装细节**
  - 构造函数接收 gl 上下文、顶点/片元源码字符串。内部完成编译、链接、错误检查。
  - 编译失败时，通过 `gl.getShaderInfoLog` 获取错误信息，抛出**带行号标注的源码**的异常（把源码按行拆开、加行号后拼进错误消息——GLSL 报错只给行号，这个小投入会节省你数小时）。
  - `use()` 内部调用 `gl.useProgram`，配合状态缓存（记录当前 Program），仅在变化时真正调用。
  - 提供 `setUniform1f`、`setUniform3fv`、`setUniformMatrix4fv` 等方法，避免外部直接操作 location。
* **Uniform 位置缓存**：链接成功后一次性查询所有 active uniform 并存入 `Map<string, WebGLUniformLocation>`，运行时零查询。
  *引导问题【Q2-5】：`gl.getUniformLocation` 为什么不该在渲染循环中调用？对不存在的 uniform 它返回什么？「明明声明了 uniform 却拿不到 location」的头号原因是什么？*

### 步骤 2：Phong 光照着色器

* **选择片元级光照（Phong Shading）**
  顶点着色器向片元着色器传递世界空间坐标和法线，片元着色器计算漫反射与镜面高光，产生光滑的高光边缘，避免 Gouraud 着色的马赫带效应。
* **逻辑拆解（片元着色器内的计算链）**
  1. `N = normalize(v_normal)` —— *引导问题【Q2-6】：法线在顶点着色器已归一化，为什么插值到片元后必须重新归一化？*
  2. `L = normalize(lightPos - worldPos)`（点光源）或常量方向（方向光，更便宜，推荐起步）。
  3. 漫反射：`max(dot(N, L), 0.0)` —— *引导问题【Q2-7】：为什么必须 clamp 到 0？背光面如果出现负光会怎样？*
  4. 镜面反射：`pow(max(dot(N, H), 0.0), shininess)`，H 为半程向量 `normalize(L + V)`（Blinn-Phong，比反射向量法更稳定且便宜）。
  5. 合成：`ambient + diffuse + specular`，注意环境光给个小常数防止背光全黑。
* **顶点着色器输入**（面向实例化预留）：
  - 顶点属性：`a_position`、`a_normal`
  - 实例属性：`i_modelMatrix`（4×4 矩阵，占 4 个属性槽，阶段三启用）
  - Uniform：`u_projection`、`u_view`、`u_lightDir`、`u_lightColor`、`u_cameraPos`、`u_time`
* **法线矩阵的工程取舍**
  教科书做法是 `mat3(transpose(inverse(model)))`，但在着色器中逐顶点求逆代价巨大。本项目 CPU 端矩阵只含旋转 + 平移（缩放动画在着色器内、且可设计为均匀缩放），故法线可直接用 `mat3(model)` 变换后归一化。**把这个前提写成注释**，未来若引入非均匀缩放，这里就是必须回访的位置。
* **调试技巧（重要）**
  光照不对时，把中间量直接当颜色输出：`fragColor = vec4(N * 0.5 + 0.5, 1.0)` 可视化法线；输出 `vec4(vec3(diff), 1.0)` 可视化漫反射强度。养成「可视化中间变量」的习惯——这是片元着色器世界里的 `console.log`。

### 步骤 3：几何体单例与 VAO

* `CubeGeometry` 职责：
  - 生成单位立方体（边长 1，中心在原点）的 24 顶点位置 + 法线数据（每面 4 顶点独立，因为法线不共享），以及 36 索引。
  - *引导问题【Q2-8】：为什么立方体不能只用 8 个顶点？只用 8 个会得到什么视觉效果？*
  - 创建 VAO，绑定位置/法线 VBO（分离或交错均可，交错对缓存更友好），配置属性指针，解绑 VAO。
  - 提供 `bind()` 方法仅调用 `gl.bindVertexArray(vao)`。
* **VAO 心智模型**：VAO 是「属性配置的快照容器」——它记录的是 `vertexAttribPointer`/`enableVertexAttribArray`/ELEMENT_ARRAY_BUFFER 绑定，而**不记录** ARRAY_BUFFER 的绑定本身。
  *自检问题【Q2-9】：解绑 VAO 后再调用 `vertexAttribPointer` 会污染谁？由此推出属性配置必须在什么时机完成？*
* **全局共享**：整个应用只创建一个 `CubeGeometry` 实例，所有实例化绘制共享同一 VAO。

## 阶段三：高性能实例化渲染与无限世界引擎（Instance & World Engine）

本阶段是核心难点。建议拆成三个独立可验证的里程碑：① 固定 N×N×N 立方体阵列的实例化渲染；② 静态的区块划分与按区块写入缓冲；③ 随摄像机移动的滑动窗口加载/卸载。

### 步骤 1：动态实例化渲染管线

* **实例属性布局**
  一个 `mat4` 占用 4 个 vec4 属性槽。着色器中：
  ```
  layout(location = 2) in vec4 i_col0;
  layout(location = 3) in vec4 i_col1;
  layout(location = 4) in vec4 i_col2;
  layout(location = 5) in vec4 i_col3;
  ```
  CPU 端配置时，对这 4 个 location 分别调用 `vertexAttribPointer`（stride = 64 字节，offset 依次 0/16/32/48）并各自设置 `vertexAttribDivisor(loc, 1)`。
  *常见陷阱清单：忘记对 4 个槽位逐一设置 divisor（只设了第一个）；stride 写成 16；实例 VBO 的属性配置发生在 VAO 解绑之后（配置丢失）；列主序数组拼装时把行当列。*
  *调试引导：先把实例矩阵全部设为单位阵 + 不同平移，画 2×2×2 = 8 个实例。8 个立方体位置正确，再上规模。*
* **动态 VBO 与 `STREAM_DRAW`**
  实例数据每帧可能改变，初始化时按**最大实例数**一次性 `bufferData(size, STREAM_DRAW)` 分配，之后每帧 `bufferSubData` 更新，绝不在运行期重新分配。
  *引导问题【Q3-1】：`STATIC_DRAW`/`DYNAMIC_DRAW`/`STREAM_DRAW` 对驱动而言是什么性质的信息？本项目每帧上传的字节量是多少？它会成为瓶颈吗？*
* **CPU 端写入策略**
  维护一个与 VBO 等容量的 CPU 侧 `Float32Array`（影子缓冲）。每帧把通过剔除的实例矩阵**紧密压缩**写入影子缓冲前段，再一次 `bufferSubData` 上传 `[0, count×16)` 区间，最后 `drawElementsInstanced(..., count)`。
  *引导问题【Q3-2】：为什么“压缩写入 + 单次上传”优于“按区块多次 bufferSubData”？这个选择和步骤 2 的槽位设计如何协调？*
* **双缓冲或环形写入**（进阶认知）
  若 GPU 仍在读取缓冲时 CPU 写入，驱动可能隐式等待。可用双 VBO 交替（奇偶帧），或“孤儿化”技巧（每帧先 `bufferData(size, null)` 再写）。本项目数据量下通常无感，但要知道排查口在哪：若 Performance 面板显示 `bufferSubData` 异常耗时，先怀疑隐式同步。

### 步骤 2：动态“无限世界”网格管理器

#### 世界分块策略

* 将世界划分为边长 `CHUNK_SIZE` 的立方体区块。立方体在区块内按局部坐标 `(ix, iy, iz)` 排列，世界坐标 = 区块基址 + 局部偏移。
* 摄像机所在区块索引：
  ```
  camChunk = ( floor(camX / CHUNK_SIZE),
               floor(camY / CHUNK_SIZE),
               floor(camZ / CHUNK_SIZE) )
  ```
  *陷阱：负坐标下 `Math.floor(-0.5) = -1` 是正确语义，但若你用 `(x / s) | 0`（截断取整）会在负半轴错位一格，导致区块边界在原点两侧不对称。务必用 floor。*
* **活跃半径** R：活跃区块集合为满足 `max(|cx-camCx|, |cy-camCy|, |cz-camCz|) ≤ R` 的所有 (cx, cy, cz)——切比雪夫距离的立方体窗口，共 `(2R+1)³` 个。
  *引导问题【Q3-3】：为什么用切比雪夫距离（立方体窗口）而非欧氏距离（球形窗口）？改用球形能省多少区块？*

#### 滑动窗口算法（逻辑拆解）

* 数据结构：
  - `activeChunks: Map<string, slotIndex>`，键为 `"cx,cy,cz"`。
  - `slotFreeList: number[]`，空闲槽位栈。
  - 每槽位对应 CPU 影子缓冲中一段长度为 `CHUNK_SIZE³ × 16` float 的固定区间。
* 每次摄像机**跨越区块边界**时（注意：不需要每帧）执行 `update(camChunk)`：
  1. 计算理想集合 `idealSet`（当前窗口内所有区块键）。
  2. **先回收**：遍历 `activeChunks`，不在 `idealSet` 中的，槽位压回 `slotFreeList`，从 Map 删除。
  3. **后分配**：遍历 `idealSet`，不在 `activeChunks` 中的，从 `slotFreeList` 弹出槽位，调用区块生成器填充该槽位的影子缓冲区间，加入 Map。
  *引导问题：*
  1. *【Q3-4】为什么必须“先回收后分配”？用一个摄像机沿对角线跨区块的具体例子论证。*
  2. *【Q3-5】摄像机一帧内跨越多个区块（高速移动或卡顿后大 deltaTime）时，该算法还正确吗？为什么？*
  3. *【Q3-6】`idealSet` 每次重建是 O((2R+1)³) ≈ 343 次循环，会是瓶颈吗？若 R 增大到 8 呢？*
* **每帧 vs 跨界触发**：将「区块窗口更新」与「剔除 + 上传」解耦——前者仅在跨区块时执行，后者每帧执行（因为剔除依赖每帧变化的摄像机姿态）。
* **生成摊销（防卡顿）**
  *引导问题【Q3-7】：跨界瞬间最多新增多少个区块？一次性生成它们会不会造成单帧尖刺？如何设计摊销策略并验证其效果？*

#### 区块内容生成器（确定性）

* `generateChunk(cx, cy, cz, outSlice)`：对区块内每个局部坐标 `(ix, iy, iz)`，由世界坐标的**整数哈希**决定该位置是否放置立方体、初始旋转相位等，再拼装 mat4 写入 `outSlice`。
* *引导问题：*
  1. *【Q3-8】如何写一个「整数坐标 → [0,1) 伪随机数」的哈希且无分配？对哈希质量的核心要求是什么？*
  2. *【Q3-9】如何验证生成的确定性？设计两个测试。*
  3. *【Q3-10】若想要“地形感”而非均匀噪声（如下方实心、上方稀疏），在哈希之上叠加什么？*

#### 实例缓冲区布局与越界保护

* 最大活跃实例总数 = `(2R+1)³ × CHUNK_SIZE³`。VBO 容量按此上限分配（× 64 字节）。启动时打印这个数字与显存估算，确认量级合理。
* 绘制时 `instanceCount` 为剔除后实际写入数。写入循环中加一个开发期断言：写入指针不得越过容量上限——越界写 `Float32Array` 不报错而是静默截断，是极难排查的 bug 来源。

### 步骤 3：基于坐标的变体动画与材质扩展

* **纯 GPU 动画原则**
  每个立方体的动画完全由其**世界坐标**与**全局时间**决定，CPU 不上传任何动画状态。顶点着色器示意：
  ```
  vec3 instancePos = i_col3.xyz;                  // 从模型矩阵第4列直接取实例平移
  float wave  = sin(instancePos.x * freqX + u_time)
              * cos(instancePos.z * freqZ + u_time) * amplitude;
  float scale = 1.0 + 0.2 * sin(instancePos.y * 5.0 + u_time * 3.0);
  vec3 animatedLocal = a_position * scale + vec3(0.0, wave, 0.0);
  ```
  *引导问题【Q3-11】：波浪相位为什么必须用世界坐标而不是 `gl_InstanceID`？用后者会出现哪两类具体症状？*
* **颜色多样性**
  用世界坐标哈希生成确定性伪随机颜色：
  ```
  float h = fract(sin(dot(instancePos.xz, vec2(12.9898, 78.233))) * 43758.5453);
  vec3 baseColor = mix(colorA, colorB, h);
  ```
  *精度警示：这个经典 sin-hash 在大坐标（10⁴ 以上）会因 Float32 sin 精度劣化而出现条纹伪影。引导：把哈希输入改为「区块局部坐标 + 区块坐标的整数哈希结果」两段式，可在无限世界中保持质量——再次呼应阶段零的精度议题。*
* **避免条件分支**：动画与配色逻辑尽量用 `mix`、`step`、`smoothstep` 表达。
  *引导问题【Q3-12】：GPU 的 SIMD 执行模型中，同一 warp 内分支发散的代价是什么？什么样的 `if` 实际无害？*
* **雾效（强烈建议，一举两得）**
  在片元着色器按视距混合雾色：`mix(color, fogColor, smoothstep(fogStart, fogEnd, dist))`，把 `fogEnd` 设在活跃窗口边缘略内侧。
  *引导问题【Q3-13】：雾效在本项目里解决的核心体验问题是什么？fogEnd 与 R·CHUNK_SIZE 应满足什么关系？背景清屏色应取什么值？*

## 阶段四：漫游摄像机系统与 CPU 端裁剪交互

### 步骤 1：欧拉角摄像机

* **球坐标映射**
  偏航角 yaw 绕世界 Y 轴，俯仰角 pitch 绕局部 X 轴。方向向量：
  ```
  dir.x = cos(pitch) * sin(yaw)
  dir.y = sin(pitch)
  dir.z = cos(pitch) * cos(yaw)
  ```
  此公式假设 yaw=0 指向 +Z、yaw 增加朝 +X 旋转。各引擎约定不同，**选定一种并在 Camera 类头部注释声明**，之后所有输入符号、lookAt 调用与之对齐。
  *自检：yaw=0, pitch=0 时 dir 应为 (0,0,1)；yaw=π/2 时应为 (1,0,0)。写成测试。*
* **万向节锁防护**
  钳位 `pitch ∈ [-π/2 + ε, π/2 - ε]`（如 ±1.55 rad）。
  *引导问题【Q4-1】：不钳位时第一帧异常出现在哪个具体函数的哪一步？从「屏幕黑了」反向推出完整的故障链路。*
* **移动模型（逻辑拆解）**
  1. 由按键状态合成**局部意图向量**（前后 + 左右 + 上下，各 ∈ {-1,0,1}）。
  2. 归一化意图向量——*否则斜向移动速度是 √2 倍，这是 FPS 游戏的经典 bug*。
  3. 把意图向量从局部坐标旋转到世界坐标：前后用方向向量的**水平投影**（去掉 y 后归一化，保持行走水平），左右用 right 向量。
  4. 速度模型：目标速度 = 意图 × maxSpeed；当前速度向目标速度指数趋近（`v += (target - v) * (1 - exp(-k·dt))`），获得平滑加減速。
  5. 位置更新必须乘 `deltaTime`。
  *引导问题【Q4-2】：为什么指数趋近形式比 `v += (target - v) * k * dt` 更稳健？在什么条件下后者会发散？*

### 步骤 2：鼠标交互与指针锁定

* **Pointer Lock API**
  点击画布调用 `canvas.requestPointerLock()`；锁定后 `mousemove` 的 `movementX/Y` 提供相对位移。
  `yaw += movementX * sensitivity; pitch -= movementY * sensitivity;`（注意 Y 取负与否取决于你想要「飞行模拟」还是「FPS」手感），随后钳位 pitch。
* **状态完整性（工程细节）**
  - 监听 `pointerlockchange`：用户按 Esc 退出锁定时，UI 应提示“点击恢复控制”，且键鼠输入应停止驱动摄像机。
  - 失焦（`blur` / `visibilitychange`）时**清空按键状态字典**。
  - *引导问题【Q4-3】：失焦时不清空按键状态会发生什么？从浏览器事件模型层面解释根因。*
* **输入与仿真解耦**：事件回调只写状态（按键字典、累积的鼠标位移），摄像机数值更新统一发生在每帧主循环中。
  *引导问题【Q4-4】：mousemove 一帧可能触发多次，「在回调中直接改摄像机」与「在主循环中统一处理」行为有何差异？后者好在哪里？*

### 步骤 3：相机半径 2 范围内的 CPU 端距离剔除

* **剔除集成位置**
  在每帧「影子缓冲 → 压缩写入」阶段（阶段三步骤 1），对每个候选实例计算到摄像机的**距离平方**，`distSq < 4.0` 则跳过写入，写入计数器不增。
  *引导问题【Q4-5】：距离剔除应该发生在区块生成时还是每帧写入时？为什么？这与「区块槽位数据 / 每帧上传缓冲」的分离设计有什么关系？*
* **为什么不在着色器 `discard`？**
  - `discard` 破坏 Early-Z 优化：片元着色器必须先执行完才能决定是否丢弃，且深度写入延迟影响后续遮挡判定。
  - CPU 端剔除直接减少实例数，顶点处理压力同步下降。
  - *引导问题【Q4-6】（延伸）：既然 discard 有害，为什么透明植被渲染又普遍使用 discard？由此提炼「在管线哪一级剔除」的决策框架。*
* **平方距离比较**：只算 `dx²+dy²+dz²` 与 `4.0` 比较，全程无 `sqrt`。剔除半径可做成可调参数便于调试，但本例硬需求为 2。

### 步骤 4：视锥剔除（从数学库到引擎的闭环）

* **平面提取（亲手推导）**
  设 `M = P · V`（投影乘视图）。裁剪条件 `-w ≤ x' ≤ w` 等价于六个半空间判断，每个平面系数可由 M 的行向量加减得到：左平面 = 第 4 行 + 第 1 行，右平面 = 第 4 行 − 第 1 行，以此类推（注意列主序数组中“取行”的下标映射——这正是阶段一那张映射表的回报时刻）。提取后归一化平面法线。
* **剔除粒度决策**
  *引导问题【Q4-7】：视锥剔除应该逐实例（数万次点-平面测试）还是逐区块（数百次 AABB-平面测试）？AABB 与平面的快速测试（p-vertex 技巧）如何做？*
* **正确性验证方法**
  把剔除结果可视化：按一个调试键冻结视锥（不再随摄像机更新），然后移动摄像机从侧面观察——被冻结视锥剔除的区块应肉眼可见地消失在锥体之外。**不做这个验证，你永远不确定剔除是“正确地省了工作”还是“错误地砍了画面”。**

### ✅ 阶段四验收标准

- [ ] WASD + 鼠标可流畅漫游；斜向移动速度与正向一致；切窗口再回来无“幽灵移动”。
- [ ] 垂直仰视/俯视极限处画面稳定无翻转、无 NaN 黑屏。
- [ ] 摄像机贴近立方体群时，半径 2 内的立方体消失，且帧率随之略升（用统计数字证实，而非感觉）。
- [ ] 冻结视锥调试模式下，剔除边界与锥体几何吻合。
- [ ] 屏幕角落显示实时统计：FPS、活跃区块数、剔除前/后实例数——这个小 HUD 是后续一切调优的仪表盘。

## 阶段五：工程化重构与数学文档规范

### 步骤 1：架构边界审查

* **纯函数数学库检查点**
  - 所有导出函数无副作用？不修改输入参数，结果严格写入 `out`？
  - 有无隐藏的模块级可变状态？暂存器池是否外部不可见、契约有文档？
  - 单例几何体、着色器程序实例是否构造后不可变？
* **控制类职责单一**
  - `Camera` 只管理位置与朝向，产出方向向量与视图矩阵，不涉及渲染。
  - `WorldManager` 只负责区块生命周期与实例数据生成，**不持有 gl、不调用任何 GL 函数**——它的产出是 `Float32Array` 与计数。
  - `Renderer` 消费上述数据执行 WebGL 调用。
  - *审查方法引导：grep 一下 `gl.` 出现在哪些文件——只允许出现在 `core/` 与 `render/`。把这条做成 lint 规则或 CI 检查更佳。*
* **禁止过度抽象**：不要为了“类”而类。一组函数只操作同一数据结构、无多态需求时，模块级函数优于类。
  *引导问题【Q5-1】（反向）：本项目中哪里反而值得抽象成类？判断「值得/不值得抽象」的统一标准是什么？*

### 步骤 2：自解释代码与“银行金库”式注释

* **注释原则**
  - 不写 `// add two vectors`，代码本身应可读。
  - 必须写“为何如此”：
    - 投影矩阵 `m[11] = -1` 的来源与推导出处。
    - 区块更新为何**先回收再分配**（峰值槽位占用论证）。
    - 为何选 `STREAM_DRAW`、为何单次压缩上传。
    - 动画频率常数的视觉选型依据；哈希函数的精度边界（坐标 > 10⁴ 时的劣化）。
    - 法线矩阵简化的前提条件（“仅当模型矩阵无非均匀缩放”——未来违反前提时这条注释就是保险丝）。
* **数学推导附录**
  在项目文档中留下透视投影、LookAt、视锥平面提取的手写推导（扫描件或 LaTeX）。未来自己或他人重构时可快速理解原始意图——知识沉淀的价值在事故复盘和二次开发时兑现。

### 步骤 3：性能验证方法论（量化，拒绝“感觉流畅”）

* **工具链**
  - Chrome Performance 面板：帧时间分布、长任务、GC 事件。重点看 p95/p99 帧时间而非平均值——卡顿是尾部现象。
  - Memory 面板：漫游 5 分钟的堆快照对比，验证稳态零分配承诺。
  - Spector.js（或同类 WebGL 调试扩展）：捕获单帧全部 GL 调用，确认 draw call 数、状态切换次数、上传字节量与设计一致。
* **建立基准再优化**
  *引导问题【Q5-2】：把 R 从 3 升到 5，实例数增长多少倍？如何设计实验区分瓶颈在顶点、片元还是 CPU 写入？*
* **回归防线**：把「漫游 60 秒，p99 帧时间 < 20 ms、内存增长 < 1 MB」写成可重复执行的手动验收脚本（或自动化），每次重构后跑一遍。

## 附录 A：黑屏/异常排查清单（按命中率排序)

图形编程的反馈是二值的——要么有画面，要么一片漆黑。按以下顺序排查可覆盖绝大多数情况：

1. **控制台有无 GL 错误 / 着色器编译错误？**（最先看，信息量最大）
2. **clear 颜色改成显眼的品红色**——能看到品红说明管线在跑，问题在绘制；纯黑可能连 clear 都没执行。
3. **检查 NaN 链路**：在 CPU 端打印视图矩阵，任何 NaN → 查 normalize 零向量、pitch 越界、未初始化的 out。
4. **去掉投影与视图矩阵**（着色器里直接输出 `vec4(a_position, 1.0)`）——能看到扭曲的几何说明顶点数据没问题，问题在矩阵。
5. **检查矩阵的 transpose 参数与列主序拼装**：`uniformMatrix4fv` 的 transpose 在 WebGL 中必须为 false，列主序数组顺序是否正确。
6. **绕序与剔除**：临时关闭 `CULL_FACE` 看物体是否出现。
7. **深度测试**：忘记 `clear(DEPTH_BUFFER_BIT)` 会导致第二帧起全部被“昨天的深度”遮挡。
8. **实例化专项**：divisor 是否对全部 4 个矩阵属性槽设置；instanceCount 是否大于 0；属性 location 是否与 layout 声明一致。
9. **near/far 与摄像机初始位置**：摄像机出生在几何体内部、或目标在 near 之内，看起来也是“黑屏”。

## 附录 C：进阶挑战（完成主线后选做）

按「收益/难度比」排序：

1. **球形活跃窗口**：切比雪夫窗口改欧氏距离，省 ≈ 48% 区块。
2. **生成摊销队列**：若主线未做，此项优先——它直接消灭最可感知的卡顿。
3. **Reversed-Z**：翻转深度映射 + `GREATER` 深度测试，远景 Z-fighting 实质性改善。
4. **浮动原点**：突破 Float32 精度墙，让“无限”在 10⁶ 坐标外依然成立（阶段 0.2 的伏笔）。
5. **Web Worker 区块生成**：得益于 `WorldManager` 不依赖 GL 的设计，可将生成移入 Worker，用 `Transferable` 传回 `Float32Array`，主线程零生成开销。
6. **遮挡剔除 / 区块内部面消隐**：相邻立方体共面不可见，生成阶段跳过被包围的立方体，实例数可降一个量级——这是从“渲染优化”走向“数据优化”的思维跃迁。
7. **LOD**：远处区块用合并的大立方体或更低实例密度表达。
8. **阴影映射**：方向光 + 深度纹理渲染到帧缓冲，接触 WebGL 2.0 的 FBO 与 sampler 比较模式。

## 结语

这份指南为你铺设了一条从底层数学到无限世界架构的完整路径。请记住三条贯穿始终的原则：

1. **每个阶段先达成最小可见成果，再叠加复杂度**——图形 bug 的定位成本随不确定层数指数增长。
2. **凡推导必动手，凡优化必量化**——纸上推导建立“正确答案长什么样”的直觉，仪表数据替代“感觉变快了”。
3. **架构约束（纯函数、单向依赖、确定性生成）不是教条**，它们分别兑换成了可测试性、可移植性（Worker 化）与“无限”本身的可实现性。

当你完整走完这些阶段，你获得的不只是一个高性能的无限三维世界，更是对 WebGL 2.0 图形管线、实时渲染优化与前端工程化的成体系理解。祝探索愉快！
